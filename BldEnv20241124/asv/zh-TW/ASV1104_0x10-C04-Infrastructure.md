# C4 基礎設施、配置與部署安全

## 控制目標

AI基礎設施必須透過安全配置、運行時隔離、可信的部署管道以及全面監控來加強防護，以抵禦特權升級、供應鏈篡改和橫向移動。只有經授權且經驗證的基礎設施組件和配置，才會通過受控流程進入生產環境，確保安全性、完整性和可審計性。

核心安全目標：只有通過加密簽名並經過漏洞掃描的基礎設施組件，才能通過自動化驗證管道進入生產環境，該管道強制執行安全策略並維護不可變的審計記錄。

## C4.1 執行時環境隔離

透過核心級隔離原語及強制存取控制，防止容器逃逸及權限提升。

|   #   | Description                                                             | Level | Role |
| :---: | ----------------------------------------------------------------------- | :---: | :--: |
| 4.1.1 | 確認所有 AI 容器都移除除了 CAP_SETUID、CAP_SETGID 以及安全基準文件中明確要求的能力之外的所有 Linux 能力。   |   1   | D/V  |
| 4.1.2 | 驗證 seccomp 配置檔阻止除預先批准的允許清單中所有系統呼叫外的所有系統呼叫，違規行為將終止容器並產生安全警報。             |   1   | D/V  |
| 4.1.3 | 驗證 AI 工作負載是否以唯讀根檔案系統運行，使用 tmpfs 存放臨時資料，並且對持久資料使用具 noexec 掛載選項的命名卷。      |   2   | D/V  |
| 4.1.4 | 驗證基於 eBPF 的運行時監控（如 Falco、Tetragon 或同等工具）能夠偵測特權提升嘗試，並在組織回應時間要求內自動終止違規程序。 |   2   | D/V  |
| 4.1.5 | 驗證高風險 AI 工作負載是否在硬體隔離環境中執行（如 Intel TXT、AMD SVM 或專用裸機節點），並進行證明驗證。         |   3   | D/V  |

## C4.2 安全建置與部署管線

透過可重現的建置和簽署的工件，確保密碼學完整性及供應鏈安全。

|   #   | Description                                                                            | Level | Role |
| :---: | -------------------------------------------------------------------------------------- | :---: | :--: |
| 4.2.1 | 確認基礎設施即代碼在每次提交時均使用工具（tfsec、Checkov 或 Terrascan）進行掃描，並阻止包含關鍵（CRITICAL）或高（HIGH）嚴重性問題的合併。 |   1   | D/V  |
| 4.2.2 | 驗證容器構建在不同構建中具有相同的 SHA256 雜湊值，以確保可重現性，並生成使用 Sigstore 簽署的 SLSA 第 3 級來源證明。                |   1   | D/V  |
| 4.2.3 | 驗證容器映像檔嵌入 CycloneDX 或 SPDX 軟體組件物件描述檔 (SBOM)，並在推送至註冊中心前使用 Cosign 進行簽署，未簽署的映像檔在部署時予以拒絕。  |   2   | D/V  |
| 4.2.4 | 驗證 CI/CD 管道使用來自 HashiCorp Vault、AWS IAM 角色或 Azure 管理身份的 OIDC 令牌，且其有效期限不超過組織安全政策的限制。    |   2   | D/V  |
| 4.2.5 | 在容器執行之前，請確認在部署過程中已驗證 Cosign 簽名和 SLSA 憑證，且驗證錯誤會導致部署失敗。                                  |   2   | D/V  |
| 4.2.6 | 驗證建置環境是否在具備無持久性儲存以及與生產虛擬私有雲（VPC）網路隔離的臨時容器或虛擬機器中運行。                                     |   2   | D/V  |

## C4.3 網路安全與存取控制

實施零信任網路，採用預設拒絕政策和加密通訊。

|   #   | Description                                                                             | Level | Role |
| :---: | --------------------------------------------------------------------------------------- | :---: | :--: |
| 4.3.1 | 驗證 Kubernetes 的 NetworkPolicies 或任何等效方案是否實現了預設拒絕的進出流量策略，並對必要的埠（如 443、8080 等）設定了明確的允許規則。 |   1   | D/V  |
| 4.3.2 | 確認是否封鎖 SSH（埠號 22）、RDP（埠號 3389）及雲端元資料端點（169.254.169.254），或是否需要基於憑證的身份驗證。                 |   1   | D/V  |
| 4.3.3 | 確認出口流量是否透過 HTTP/HTTPS 代理（Squid、Istio 或雲端 NAT 閘道）進行過濾，並使用網域允許清單，同時記錄被阻擋的請求。              |   2   | D/V  |
| 4.3.4 | 驗證跨服務通訊是否使用雙向 TLS，且憑證依照組織政策進行輪替並強制執行憑證驗證（不允許跳過驗證標誌）。                                    |   2   | D/V  |
| 4.3.5 | 驗證 AI 基礎架構是否在專用 VPC/VNet 中運行，且無直接的網際網路存取，僅透過 NAT 網關或堡壘主機進行通訊。                           |   2   | D/V  |

## C4.4 秘密資料與密碼金鑰管理

| 4.4.6 | 確認機密資訊是透過 Kubernetes 秘密（secrets）、掛載的卷（mounted volumes）或初始化容器（init containers）注入，並確保機密資訊絕不嵌入環境變數或映像檔中。       |   2   | D/V  |

## C4.5 AI 工作負載沙箱測試與驗證

將不受信任的人工智慧模型隔離於安全沙盒中，並進行全面行為分析。

|   #   | Description                                                                                                          | Level | Role |
| :---: | -------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 4.5.1 | 確認外部 AI 模型在 gVisor、微型虛擬機（如 Firecracker、CrossVM）或 Docker 容器中執行，且使用 --security-opt=no-new-privileges 和 --read-only 參數。 |   1   | D/V  |
| 4.5.2 | 確認沙盒環境沒有網路連接（--network=none）或僅有本地主機存取，並且所有外部請求均被 iptables 規則阻擋。                                                      |   1   | D/V  |
| 4.5.3 | 確保AI模型驗證包括組織定義的測試覆蓋範圍及行為分析的自動化紅隊測試，以檢測後門。                                                                            |   2   | D/V  |
| 4.5.4 | 確認在 AI 模型被推廣至生產環境之前，其沙盒結果已由授權的安全人員進行密碼簽名，並存儲在不可變的審計日誌中。                                                              |   2   | D/V  |
| 4.5.5 | 確認沙盒環境在每次評估之間，均從黃金映像重新建立，並且完成檔案系統及記憶體的徹底清理。                                                                          |   2   | D/V  |
