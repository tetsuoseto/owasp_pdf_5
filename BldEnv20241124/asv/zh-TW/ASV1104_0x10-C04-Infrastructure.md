# C4 基礎架構、配置與部署安全性

## 控制目標

人工智慧基礎設施必須透過安全配置、執行隔離、受信任的部署管線和全面監控來加強防範特權提升、供應鏈篡改及橫向移動。只有經授權且經驗證的基礎設施元件與配置，才能透過維持安全性、完整性及可稽核性的受控流程抵達生產環境。

核心安全目標：只有經過加密簽名並通過漏洞掃描的基礎設施元件，才能透過自動化驗證管線進入生產環境，該管線強制執行安全政策並維護不可變的審計追蹤。

## C4.1 執行環境隔離

通過核心層級的隔離原語和強制訪問控制來防止容器逃脫和權限提升。

|   #   | Description                                                             | Level | Role |
| :---: | ----------------------------------------------------------------------- | :---: | :--: |
| 4.1.1 | 確認所有 AI 容器除了 CAP_SETUID、CAP_SETGID 以及安全基線中明確要求的能力外，皆移除所有 Linux 能力。      |   1   | D/V  |
| 4.1.2 | 驗證 seccomp 配置檔是否封鎖所有系統呼叫，僅允許預先核准的允許清單中的系統呼叫，違規行為將導致容器終止並產生安全警報。         |   1   | D/V  |
| 4.1.3 | 確認 AI 工作負載以唯讀根檔案系統、用於臨時資料的 tmpfs，以及強制使用 noexec 掛載選項的命名卷來執行持久性資料。        |   2   | D/V  |
| 4.1.4 | 驗證基於 eBPF 的運行時監控（Falco、Tetragon 或等效工具）是否能檢測到權限提升嘗試，並在組織響應時間要求內自動終止違規進程。 |   2   | D/V  |
| 4.1.5 | 確認高風險的 AI 工作負載在硬體隔離環境中執行（如 Intel TXT、AMD SVM 或專用的裸機節點），並進行鑑證驗證。         |   3   | D/V  |

## C4.2 安全建置與部署管線

透過可重現的建置與簽署的工件，確保密碼學完整性與供應鏈安全。

|   #   | Description                                                                        | Level | Role |
| :---: | ---------------------------------------------------------------------------------- | :---: | :--: |
| 4.2.1 | 驗證基礎設施即程式碼是否在每次提交時都使用工具（tfsec、Checkov 或 Terrascan）進行掃描，並在發現重大或高嚴重性問題時阻止合併。         |   1   | D/V  |
| 4.2.2 | 驗證容器構建在多次構建中是否具有可重現的相同 SHA256 雜湊，並生成使用 Sigstore 簽署的 SLSA 等級 3 證明文件。                |   1   | D/V  |
| 4.2.3 | 確認容器映像在推送到註冊中心之前，已嵌入 CycloneDX 或 SPDX SBOM，並使用 Cosign 簽名，未簽名的映像在部署時予以拒絕。           |   2   | D/V  |
| 4.2.4 | 確認 CI/CD 管道使用來自 HashiCorp Vault、AWS IAM 角色或 Azure 管理身分的 OIDC 令牌，其有效期限不超過組織安全政策的限制。 |   2   | D/V  |
| 4.2.5 | 確認在容器執行前的部署過程中，Cosign 簽名和 SLSA 來源均已被驗證，且驗證錯誤會導致部署失敗。                               |   2   | D/V  |
| 4.2.6 | 驗證建置環境是否運行於臨時容器或虛擬機器中，且無持久性存儲並與生產VPC進行網路隔離。                                        |   2   | D/V  |

## C4.4 秘密與密碼學金鑰管理

| 4.4.2 | 驗證加密金鑰是否在符合 FIPS 140-2 Level 2 的硬體安全模組（HSM）（如 AWS CloudHSM、Azure Dedicated HSM）中生成，並依照組織的加密政策進行金鑰輪替。      |   1   | D/V  |
| 4.4.3 | 驗證秘密輪換是否已實現自動化，並具備零停機部署及由人員變動或安全事件即時觸發的輪換機制。                                                              |   2   | D/V  |
| 4.4.4 | 確認容器映像檔已使用工具（GitLeaks、TruffleHog 或 detect-secrets）進行掃描，並阻止包含 API 金鑰、密碼或憑證的建置。                             |   2   | D/V  |
| 4.4.5 | 確認生產環境的秘密訪問需要使用帶有硬體代幣（YubiKey、FIDO2）的多重身份驗證（MFA），並且此過程由包含用戶身份和時間戳記的不可變審計日誌記錄。                             |   2   | D/V  |
| 4.4.6 | 確認機密資訊是透過 Kubernetes 機密、掛載的卷或初始化容器注入，並確保機密資訊絕不會內嵌在環境變數或映像檔中。                                              |   2   | D/V  |

## C4.5 AI 工作負載沙盒與驗證

| 4.5.4 | 確認在AI模型推向生產環境之前，其沙盒結果已由授權的安全人員進行密碼簽名，並存儲在不可變的審計日誌中。                                                                      |   2   | D/V  |
| 4.5.5 | 確認沙盒環境在每次評估之間均從黃金映像重建，並完成完整的檔案系統及記憶體清理。                                                                                  |   2   | D/V  |
