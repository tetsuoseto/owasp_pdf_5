# 10 對抗魯棒性與隱私防禦

## 控制目標

確保在面對規避、推理、提取或中毒攻擊時，AI 模型保持可靠、保護隱私並抗拒濫用。

---

## 10.1 模型對齊與安全

防範有害或違反政策的輸出。

|   #    | Description                                 | Level | Role |
| :----: | ------------------------------------------- | :---: | :--: |
| 10.1.1 | 確認對齊測試套件（紅隊提示、越獄探測、禁止內容）進行版本控制，並在每次模型發布時執行。 |   1   | D/V  |
| 10.1.2 | 確認拒絕機制與安全完成防護措施已被強制執行。                      |   1   |  D   |
| 10.1.3 | 驗證自動評估器是否測量有害內容率並標記超出設定閾值的回歸。               |   2   | D/V  |
| 10.1.4 | 驗證反越獄訓練是否有文件記錄且可重現。                         |   2   |  D   |
| 10.1.5 | 驗證正式政策合規性證明或認證監控是否涵蓋關鍵領域。                   |   3   |  V   |

---

## 10.2 對抗樣本強化

提高對被操縱輸入的韌性。穩健的對抗訓練和基準評分是目前的最佳實踐。

|   #    | Description                       | Level | Role |
| :----: | --------------------------------- | :---: | :--: |
| 10.2.1 | 驗證專案儲存庫是否包含具有可重現隨機種子的對抗訓練配置。      |   1   |  D   |
| 10.2.2 | 驗證對抗樣本檢測是否會在生產流程中觸發阻擋警報。          |   2   | D/V  |
| 10.2.4 | 驗證經認證的魯棒性證明或區間界限證明是否涵蓋至少最重要的關鍵類別。 |   3   |  V   |
| 10.2.5 | 確認迴歸測試使用自適應攻擊來證實沒有可測量的魯棒性損失。      |   3   |  V   |

---

## 10.3 成員推斷緩解措施

限制判斷一筆記錄是否在訓練資料中的能力。差分隱私和置信度分數遮罩仍然是目前已知最有效的防禦方法。

|   #    | Description                               | Level | Role |
| :----: | ----------------------------------------- | :---: | :--: |
| 10.3.1 | 驗證每次查詢的熵正則化或溫度調節是否能減少過度自信的預測。             |   1   |  D   |
| 10.3.2 | 驗證訓練過程對敏感資料集採用ε界限的差分隱私優化方法。               |   2   |  D   |
| 10.3.3 | 驗證攻擊模擬（影子模型或黑盒模型）在未使用的資料上顯示攻擊 AUC ≤ 0.60。 |   2   |  V   |

---

## 10.4 模型反演抗性

防止重建私人屬性。近期調查強調輸出截斷與差分隱私（DP）保證作為實際防禦措施。

|   #    | Description                     | Level | Role |
| :----: | ------------------------------- | :---: | :--: |
| 10.4.1 | 確認敏感屬性絕不會直接輸出；在必要時，使用區間分組或單向轉換。 |   1   |  D   |
| 10.4.2 | 驗證查詢速率限制是否會對來自同一主體的重複自適應查詢進行節流。 |   1   | D/V  |
| 10.4.3 | 確認模型是否以隱私保護噪聲進行訓練。              |   2   |  D   |

---

## 10.5 模型提取防禦

偵測並阻止未經授權的複製。建議使用浮水印和查詢模式分析。

|   #    | Description                                             | Level | Role |
| :----: | ------------------------------------------------------- | :---: | :--: |
| 10.5.1 | 驗證推論閘道確實執行全球及每個 API 金鑰的速率限制，並依據模型的記憶閾值進行調整。             |   1   |  D   |
| 10.5.2 | 驗證查詢熵和輸入複數性統計是否用於自動化提取檢測器。                              |   2   | D/V  |
| 10.5.3 | 驗證脆弱或機率性浮水印能夠在對疑似克隆進行不超過 1,000 次查詢時，以 p < 0.01 的水準得到證明。 |   2   |  V   |
| 10.5.4 | 確認水印金鑰和觸發集存儲在硬體安全模組中，並每年進行輪換。                           |   3   |  D   |
| 10.5.5 | 驗證抽取警示事件是否包含違規查詢，並且已整合至事件響應劇本中。                         |   3   |  V   |

---

## 10.6 推理時期的中毒數據檢測

識別並中和帶有後門或被污染的輸入。

|   #    | Description                         | Level | Role |
| :----: | ----------------------------------- | :---: | :--: |
| 10.6.1 | 確認輸入在模型推論前通過異常檢測器（例如 STRIP、一致性評分）。  |   1   |  D   |
| 10.6.2 | 確認檢測器閾值已在乾淨/受污染的驗證集上調整，以達成低於5%的誤報率。 |   1   |  V   |
| 10.6.3 | 驗證被標記為中毒的輸入是否會觸發軟阻擋和人工審查流程。         |   2   |  D   |
| 10.6.4 | 驗證檢測器是否經過自適應、無觸發後門攻擊的壓力測試。          |   2   |  V   |
| 10.6.5 | 確認檢測效能指標已被記錄，並定期使用最新威脅情報重新評估。       |   3   |  D   |

---

## 10.7 動態安全政策調整

基於威脅情報和行為分析的即時安全策略更新。

|   #    | Description                           | Level | Role |
| :----: | ------------------------------------- | :---: | :--: |
| 10.7.1 | 驗證安全政策能夠在不重新啟動代理的情況下動態更新，同時保持政策版本完整性。 |   1   | D/V  |
| 10.7.2 | 驗證政策更新是否由授權的安全人員以密碼學方式簽署，並在應用前進行驗證。   |   2   | D/V  |
| 10.7.3 | 確認動態政策更改有完整的審計追蹤記錄，包括理由、批准鏈和回滾程序。     |   2   | D/V  |
| 10.7.4 | 驗證自適應安全機制是否根據風險情境和行為模式調整威脅偵測的敏感度。     |   3   | D/V  |
| 10.7.5 | 驗證策略調整決策具可解釋性，並包含供安全團隊審查的證據追蹤。        |   3   | D/V  |

---

## 10.8 基於反射的安全分析

透過代理自我反思與元認知分析進行安全驗證。

|   #    | Description                        | Level | Role |
| :----: | ---------------------------------- | :---: | :--: |
| 10.8.1 | 確認代理反思機制包含以安全為重點的決策和行動自我評估。        |   1   | D/V  |
| 10.8.2 | 驗證反射輸出是否經過驗證，以防止對抗性輸入操縱自我評估機制。     |   2   | D/V  |
| 10.8.3 | 驗證元認知安全分析是否能識別代理推理過程中的潛在偏見、操控或妥協。  |   2   | D/V  |
| 10.8.4 | 驗證基於反射的安全警告是否觸發強化監控及可能的人為干預工作流程。   |   3   | D/V  |
| 10.8.5 | 驗證從安全反思持續學習能提升威脅檢測能力，且不會降低合法功能的效能。 |   3   | D/V  |

---

## 10.9 演化與自我改進安全

可自我修改與進化的代理系統之安全控制措施。

|   #    | Description                      | Level | Role |
| :----: | -------------------------------- | :---: | :--: |
| 10.9.1 | 驗證自我修改功能僅限於指定的安全區域，並具有正式驗證邊界。    |   1   | D/V  |
| 10.9.2 | 確認進化提案在實施前進行安全影響評估。              |   2   | D/V  |
| 10.9.3 | 確認自我改進機制包含具備完整性驗證的回滾能力。          |   2   | D/V  |
| 10.9.4 | 驗證元學習安全性以防止改進演算法的對抗性操縱。          |   3   | D/V  |
| 10.9.5 | 驗證遞歸自我改進受到形式化安全約束的限制，並具備收斂的數學證明。 |   3   | D/V  |
