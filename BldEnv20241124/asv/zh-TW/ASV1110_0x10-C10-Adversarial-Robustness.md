# 10 對抗魯棒性與隱私防護

## 控制目標

確保人工智慧模型在面對規避、推理、擷取或中毒攻擊時，保持可靠性、隱私保護和抗濫用能力。

---

## 10.1 模型對齊與安全性

防範有害或違反政策的輸出。

|   #    | Description                                    | Level | Role |
| :----: | ---------------------------------------------- | :---: | :--: |
| 10.1.1 | 確認對齊測試套件（紅隊提示、越獄探針、不允許的內容）已納入版本控制，並在每次模型發佈時執行。 |   1   | D/V  |
| 10.1.2 | 確認拒絕和安全完成防護措施已被執行。                             |   1   |  D   |
| 10.1.3 | 驗證自動評估器是否測量有害內容率並標記超過設定閾值的回歸。                  |   2   | D/V  |
| 10.1.4 | 確認反越獄訓練已被記錄且可重現。                               |   2   |  D   |
| 10.1.5 | 驗證正式的政策合規證明或經認證的監控是否涵蓋關鍵領域。                    |   3   |  V   |

---

## 10.2 對抗樣本強化

提高對被操縱輸入的韌性。強健的對抗訓練和基準評分是目前的最佳實踐。

|   #    | Description                      | Level | Role |
| :----: | -------------------------------- | :---: | :--: |
| 10.2.1 | 確認專案資料庫包含具有可重現種子的對抗訓練配置。         |   1   |  D   |
| 10.2.2 | 驗證對抗樣本檢測是否在生產管線中觸發阻斷警報。          |   2   | D/V  |
| 10.2.4 | 驗證經認證的魯棒性證明或區間界證明是否涵蓋至少最重要的關鍵類別。 |   3   |  V   |
| 10.2.5 | 驗證迴歸測試是否使用自適應攻擊以確認沒有可測量的魯棒性損失。   |   3   |  V   |

---

## 10.3 成員推斷緩解措施

限制決定某筆資料是否在訓練資料中的能力。差分隱私和置信度分數遮蔽仍然是已知最有效的防禦方法。

|   #    | Description                         | Level | Role |
| :----: | ----------------------------------- | :---: | :--: |
| 10.3.1 | 驗證每查詢熵正則化或溫度調整是否能減少過度自信的預測。         |   1   |  D   |
| 10.3.2 | 驗證訓練過程是否針對敏感資料集採用 ε-界限差分隱私優化方法。     |   2   |  D   |
| 10.3.3 | 驗證攻擊模擬（影子模型或黑箱）在持出數據上的攻擊AUC ≤ 0.60。 |   2   |  V   |

---

## 10.4 模型反演抵抗

防止重建私人屬性。近期調查強調輸出截斷和差分隱私（DP）保證作為實用防禦措施。

|   #    | Description                    | Level | Role |
| :----: | ------------------------------ | :---: | :--: |
| 10.4.1 | 確認敏感屬性絕不會直接輸出；如有需要，請使用分桶或單向轉換。 |   1   |  D   |
| 10.4.2 | 驗證查詢速率限制是否會限制來自相同主體的重複自適應查詢。   |   1   | D/V  |
| 10.4.3 | 確認模型訓練時使用了具隱私保護的噪音。            |   2   |  D   |

---

## 10.5 模型提取防護

偵測並阻止未經授權的複製。建議使用浮水印和查詢模式分析。

|   #    | Description                                        | Level | Role |
| :----: | -------------------------------------------------- | :---: | :--: |
| 10.5.1 | 驗證推理閘道是否強制執行全局及每個 API 金鑰的速率限制，並調整至模型的記憶閾值。         |   1   |  D   |
| 10.5.2 | 驗證查詢熵與輸入多樣性統計數據是否用於自動提取檢測器。                        |   2   | D/V  |
| 10.5.3 | 驗證易碎或概率性水印是否能在對疑似克隆體進行不超過1,000次查詢時以p < 0.01的概率被證明。 |   2   |  V   |
| 10.5.4 | 驗證水印金鑰和觸發集是否存儲在硬體安全模組中，並且每年進行輪替。                   |   3   |  D   |
| 10.5.5 | 驗證提取警報事件是否包含違規查詢，並且已與事故響應劇本整合。                     |   3   |  V   |

---

## 10.6 推理階段的中毒數據檢測

識別並中和後門或被污染的輸入。

|   #    | Description                            | Level | Role |
| :----: | -------------------------------------- | :---: | :--: |
| 10.6.1 | 在模型推理之前，確認輸入資料通過異常檢測器（例如，STRIP、一致性評分）。 |   1   |  D   |
| 10.6.2 | 確認探測器閾值在乾淨/受污染的驗證集上進行調整，以實現低於5%的誤報率。   |   1   |  V   |
| 10.6.3 | 驗證被標記為受污染的輸入是否會觸發軟封鎖和人工審核流程。           |   2   |  D   |
| 10.6.4 | 驗證探測器是否經過適應性、無觸發後門攻擊的壓力測試。             |   2   |  V   |
| 10.6.5 | 確認檢測效能指標已被記錄，並且定期使用最新的威脅情報重新評估。        |   3   |  D   |

---

## 10.7 動態安全政策調整

基於威脅情報與行為分析的即時安全政策更新。

|   #    | Description                           | Level | Role |
| :----: | ------------------------------------- | :---: | :--: |
| 10.7.1 | 驗證安全策略能在不重新啟動代理的情況下動態更新，同時保持策略版本的完整性。 |   1   | D/V  |
| 10.7.2 | 驗證政策更新是否由授權的安全人員以密碼學方式簽署，並在應用前進行驗證。   |   2   | D/V  |
| 10.7.3 | 驗證動態政策變更是否完整記錄審計追蹤，包括變更理由、批准流程鏈及回滾程序。 |   2   | D/V  |
| 10.7.4 | 驗證自適應安全機制根據風險情境和行為模式調整威脅檢測的靈敏度。       |   3   | D/V  |
| 10.7.5 | 驗證政策調整決策具備可解釋性，並包含供安全團隊審核的證據軌跡。       |   3   | D/V  |

---

## 10.8 基於反射的安全分析

通過代理自我反思和元認知分析進行安全驗證。

|   #    | Description                       | Level | Role |
| :----: | --------------------------------- | :---: | :--: |
| 10.8.1 | 驗證代理反思機制包括針對安全的決策和行動自我評估。         |   1   | D/V  |
| 10.8.2 | 確認反射輸出已經過驗證，以防止對抗性輸入操控自我評估機制。     |   2   | D/V  |
| 10.8.3 | 驗證元認知安全分析是否能識別代理推理過程中的潛在偏見、操控或妥協。 |   2   | D/V  |
| 10.8.4 | 驗證基於反射的安全警告是否會觸發加強的監控及潛在人為干預流程。   |   3   | D/V  |
| 10.8.5 | 驗證持續從安全反思中學習能提升威脅檢測能力且不會降低合法功能。   |   3   | D/V  |

---

## 10.9 演化與自我改進安全

具備自我修改與演化能力的代理系統之安全控管。

|   #    | Description                    | Level | Role |
| :----: | ------------------------------ | :---: | :--: |
| 10.9.1 | 確認自我修改能力僅限於指定的安全區域，並具有正式驗證的邊界。 |   1   | D/V  |
| 10.9.2 | 確認演進提案在實施前經過安全影響評估。            |   2   | D/V  |
| 10.9.3 | 驗證自我改進機制包含具備完整性驗證的回滾功能。        |   2   | D/V  |
| 10.9.4 | 驗證元學習安全是否防止改進算法的對抗性操控。         |   3   | D/V  |
| 10.9.5 | 驗證遞迴自我改進受形式安全約束所限制，並附上數學收斂證明。  |   3   | D/V  |
