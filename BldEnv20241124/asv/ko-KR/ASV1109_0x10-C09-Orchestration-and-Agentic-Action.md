# 9 자율 오케스트레이션 및 행위자 행위 보안

## 통제 목표

자율 또는 다중 에이전트 AI 시스템이 명시적으로 의도되고, 인증되며, 감사 가능하고, 한정된 비용 및 위험 한계 내에서만 동작을 실행할 수 있도록 보장해야 합니다. 이는 자율 시스템 침해, 도구 오용, 에이전트 루프 감지, 통신 탈취, 신원 위조, 군집 조작, 의도 조작과 같은 위협으로부터 보호합니다.

|   #   | Description                                                                        | Level | Role |
| :---: | ---------------------------------------------------------------------------------- | :---: | :--: |

| 9.1.2 | 특권 또는 되돌릴 수 없는 작업(예: 코드 커밋, 금융 이체)이 실행되기 전에 감사 가능한 채널을 통해 명시적인 인간 승인이 필요함을 확인하십시오. |   1   | D/V  |
| 9.1.3 | 실시간 리소스 모니터가 예산 임계값을 초과할 경우 서킷 브레이커 중단을 트리거하여 추가 작업 확장을 중지하는지 확인하십시오.              |   2   |  D   |
| 9.1.4 | 회로 차단기 이벤트가 포렌식 검토를 위해 에이전트 ID, 트리거 조건 및 캡처된 계획 상태와 함께 기록되는지 확인하십시오.               |   2   | D/V  |
| 9.1.5 | 보안 테스트가 예산 소진 및 무한 실행 시나리오를 포함하는지 확인하고, 데이터 손실 없이 안전하게 중단되는지 검증하십시오.               |   3   |  V   |
| 9.1.6 | 예산 정책이 코드로서의 정책(policy-as-code)로 표현되고 CI/CD에서 구성 변이를 차단하기 위해 강제 적용되는지 확인하십시오.      |   3   |  D   |

## 9.10 에이전트 추론 전략 보안

ReAct, Chain-of-Thought, Tree-of-Thoughts 접근 방식을 포함한 다양한 추론 전략의 안전한 선택 및 실행.

|   #    | Description                                                                                                 | Level | Role |
| :----: | ----------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 9.10.1 | 추론 전략 선택이 결정론적 기준(입력 복잡성, 작업 유형, 보안 컨텍스트)을 사용하는지 확인하고 동일한 입력이 동일한 보안 컨텍스트 내에서 동일한 전략 선택을 생성하는지 검증하십시오.      |   1   | D/V  |
| 9.10.2 | 각 추론 전략(ReAct, Chain-of-Thought, Tree-of-Thoughts)에 대해 고유한 인지 접근 방식에 맞는 입력 검증, 출력 정제, 실행 시간 제한이 있는지 확인하십시오. |   1   | D/V  |
| 9.10.3 | 감사 추적 재구성을 위해 입력 특성, 선택 기준 값 및 실행 메타데이터를 포함한 완전한 컨텍스트와 함께 추론 전략 전환이 기록되는지 확인하십시오.                           |   2   | D/V  |
| 9.10.4 | 트리-오브-생각(Tree-of-Thoughts) 추론에는 정책 위반, 자원 한계 또는 안전 경계가 감지되면 탐색을 종료하는 가지 가지치기 메커니즘이 포함되어 있는지 확인하십시오.         |   2   | D/V  |
| 9.10.5 | ReAct(Reason-Act-Observe) 사이클이 각 단계에서 검증 체크포인트를 포함하는지 확인하십시오: 추론 단계 검증, 행동 승인, 그리고 진행하기 전에 관찰 데이터 정제.       |   2   | D/V  |
| 9.10.6 | 추론 전략 성능 지표(실행 시간, 자원 사용량, 출력 품질)가 구성된 임계값을 벗어날 때 자동 경고와 함께 모니터링되는지 확인하십시오.                                 |   3   | D/V  |
| 9.10.7 | 여러 전략을 결합한 하이브리드 추론 방법이 모든 구성 전략의 입력 검증과 출력 제약 조건을 준수하며 보안 통제를 우회하지 않는지 검증하십시오.                             |   3   | D/V  |
| 9.10.8 | 추론 전략 보안 테스트에는 잘못된 입력으로 퍼징, 전략 전환을 유도하는 적대적 프롬프트, 그리고 각 인지 접근법에 대한 경계 조건 테스트가 포함되어 있음을 확인하십시오.              |   3   | D/V  |

## 9.11 에이전트 라이프사이클 상태 관리 및 보안

암호화된 감사 추적과 정의된 복구 절차를 통해 에이전트 초기화, 상태 전환 및 종료를 안전하게 처리합니다.

|   #    | Description                                                                                                           | Level | Role |
| :----: | --------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 9.11.1 | 에이전트 초기화가 하드웨어 기반 자격 증명을 통한 암호학적 신원 설정과 에이전트 ID, 타임스탬프, 구성 해시 및 초기화 매개변수를 포함하는 변경 불가능한 시작 감사 로그를 포함하는지 확인하십시오.        |   1   | D/V  |
| 9.11.2 | 에이전트 상태 전환이 암호학적으로 서명되고, 타임스탬프가 찍히며, 트리거 이벤트, 이전 상태 해시, 새로운 상태 해시, 수행된 보안 검증을 포함한 완전한 컨텍스트와 함께 기록되었는지 확인하십시오.         |   2   | D/V  |
| 9.11.3 | 에이전트 종료 절차에 암호화 삭제 또는 다중 덮어쓰기를 사용한 안전한 메모리 삭제, 인증서 기관 알림을 통한 자격 증명 취소, 변조 방지 종료 인증서 생성이 포함되어 있는지 확인하십시오.              |   2   | D/V  |
| 9.11.4 | 에이전트 복구 메커니즘이 암호화 체크섬(SHA-256 이상)을 사용하여 상태 무결성을 검증하고, 데이터 손상이 감지될 경우 자동 알림 및 수동 승인 요구 사항과 함께 알려진 정상 상태로 롤백하는지 확인하십시오. |   3   | D/V  |
| 9.11.5 | 에이전트 지속성 메커니즘이 민감한 상태 데이터를 에이전트별 AES-256 키로 암호화하고, 최대 90일의 구성 가능한 일정에 따라 제로 다운타임 배포로 안전한 키 교체를 구현하는지 확인하십시오.          |   3   | D/V  |

## 9.12 도구 통합 보안 프레임워크

정의된 위험 평가 및 승인 프로세스를 포함한 동적 도구 로딩, 실행 및 결과 검증을 위한 보안 제어.

|   #    | Description                                                                                                               | Level | Role |
| :----: | ------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 9.12.1 | 도구 설명자에 필수 권한(읽기/쓰기/실행), 위험 수준(낮음/중간/높음), 자원 제한(CPU, 메모리, 네트워크) 및 도구 매니페스트에 문서화된 검증 요구사항을 명시하는 보안 메타데이터가 포함되어 있는지 확인하십시오. |   1   | D/V  |
| 9.12.2 | 도구 실행 결과가 통합 전에 예상 스키마(JSON 스키마, XML 스키마) 및 보안 정책(출력 정화, 데이터 분류)에 따라 검증되고, 제한 시간 및 오류 처리 절차가 적용되는지 확인하십시오.                |   1   | D/V  |
| 9.12.3 | 도구 상호작용 로그에 권한 사용, 데이터 접근 패턴, 실행 시간, 자원 사용량, 반환 코드 등 자세한 보안 컨텍스트가 포함되어 있으며 SIEM 통합을 위한 구조화된 로깅이 이루어져 있는지 확인하십시오.          |   2   | D/V  |
| 9.12.4 | 동적 도구 로딩 메커니즘이 PKI 인프라를 사용하여 디지털 서명을 검증하고, 실행 전에 샌드박스 격리 및 권한 검증을 포함한 보안 로딩 프로토콜을 구현하는지 확인하십시오.                           |   2   | D/V  |
| 9.12.5 | 도구 보안 평가가 새로운 버전에 대해 자동으로 시작되며, 정적 분석, 동적 테스트 및 보안 팀 검토를 포함하는 필수 승인 절차와 문서화된 승인 기준 및 SLA 요건이 있는지 확인하십시오.                  |   3   | D/V  |
