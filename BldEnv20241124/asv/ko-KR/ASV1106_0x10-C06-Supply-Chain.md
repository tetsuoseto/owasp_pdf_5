# 모델, 프레임워크 및 데이터에 대한 C6 공급망 보안

## 제어 목표

AI 공급망 공격은 제3자 모델, 프레임워크 또는 데이터 세트를 이용하여 백도어, 편향 또는 악용 가능한 코드를 삽입하는 것을 악용합니다. 이러한 제어는 전체 모델 수명 주기를 보호하기 위해 종단 간 출처 추적, 취약점 관리 및 모니터링을 제공합니다.

---

## C6.1 사전 학습 모델 검증 및 출처 확인

미세 조정이나 배포 전에 제3자 모델의 출처, 라이선스 및 숨겨진 동작을 평가하고 인증하십시오.

|   #   | Description                                                               | Level | Role |
| :---: | ------------------------------------------------------------------------- | :---: | :--: |
| 6.1.1 | 모든 서드파티 모델 아티팩트에 출처 저장소 및 커밋 해시를 식별하는 서명된 출처 기록이 포함되어 있는지 확인하십시오.         |   1   | D/V  |
| 6.1.2 | 수입 전에 자동화 도구를 사용하여 모델에 악성 레이어나 트로이 목마 트리거가 있는지 스캔하는지 확인하세요.               |   1   | D/V  |
| 6.1.3 | 전이 학습 미세 조정이 숨겨진 동작을 감지하기 위한 적대적 평가를 통과하는지 확인합니다.                         |   2   |  D   |
| 6.1.4 | 모델 라이선스, 수출 통제 태그 및 데이터 출처 진술이 ML-BOM 항목에 기록되어 있는지 확인하십시오.                |   2   |  V   |
| 6.1.5 | 고위험 모델(공개적으로 업로드된 가중치, 검증되지 않은 제작자)이 인간의 검토 및 승인 전까지 격리 상태를 유지하는지 확인하십시오. |   3   | D/V  |

---

## C6.2 프레임워크 및 라이브러리 스캐닝

런타임 스택을 안전하게 유지하기 위해 ML 프레임워크와 라이브러리를 지속적으로 CVE(공개 취약점 데이터베이스) 및 악성 코드에 대해 검사합니다.

|   #   | Description                                                 | Level | Role |
| :---: | ----------------------------------------------------------- | :---: | :--: |
| 6.2.1 | CI 파이프라인이 AI 프레임워크 및 핵심 라이브러리에 대해 의존성 스캐너를 실행하는지 확인하십시오.    |   1   | D/V  |
| 6.2.2 | 치명적인 취약점(CVSS ≥ 7.0)이 프로덕션 이미지로의 프로모션을 차단하는지 확인하십시오.        |   1   | D/V  |
| 6.2.3 | 포크되거나 벤더링된 ML 라이브러리에서 정적 코드 분석이 실행되는지 확인하십시오.               |   2   |  D   |
| 6.2.4 | 프레임워크 업그레이드 제안서에 공개 CVE 피드를 참조한 보안 영향 평가가 포함되어 있는지 확인하십시오.  |   2   |  V   |
| 6.2.5 | 서명된 SBOM에서 벗어난 예상치 못한 동적 라이브러리 로드에 대해 런타임 센서가 경고하는지 확인하십시오. |   3   |  V   |

---

## C6.3 의존성 고정 및 검증

모든 종속성을 변경 불가능한 다이제스트에 고정하고 빌드를 재현하여 동일하고 변조되지 않은 아티팩트를 보장합니다.

|   #   | Description                                               | Level | Role |
| :---: | --------------------------------------------------------- | :---: | :--: |
| 6.3.1 | 모든 패키지 관리자가 락파일을 통해 버전 고정을 강제하는지 확인하십시오.                  |   1   | D/V  |
| 6.3.2 | 컨테이너 참조에서 변경 가능한 태그 대신 변경 불가능한 해시값(다이제스트)이 사용되었는지 확인하십시오. |   1   | D/V  |
| 6.3.3 | 재현 가능한 빌드 검사에서 CI 실행 간 해시를 비교하여 동일한 출력이 생성되는지 확인합니다.      |   2   |  D   |
| 6.3.4 | 감사 추적성을 위해 빌드 증명서가 18개월 동안 저장되는지 확인하십시오.                  |   2   |  V   |
| 6.3.5 | 만료된 의존성이 자동 PR을 트리거하여 고정된 버전을 업데이트하거나 포크하는지 확인하십시오.       |   3   |  D   |

---

## C6.4 신뢰할 수 있는 출처 강제 적용

암호학적으로 검증되고 조직에서 승인한 출처에서만 아티팩트 다운로드를 허용하고 다른 모든 것은 차단하십시오.

|   #   | Description                                                  | Level | Role |
| :---: | ------------------------------------------------------------ | :---: | :--: |
| 6.4.1 | 모델 가중치, 데이터셋 및 컨테이너가 승인된 도메인 또는 내부 레지스트리에서만 다운로드되었는지 확인하십시오. |   1   | D/V  |
| 6.4.2 | Sigstore/Cosign 서명이 아티팩트가 로컬에 캐시되기 전에 게시자 신원을 검증하는지 확인하십시오.  |   1   | D/V  |
| 6.4.3 | 인증되지 않은 아티팩트 다운로드를 차단하여 신뢰할 수 있는 소스 정책을 시행하는지 퇴출 프록시가 확인합니다. |   2   |  D   |
| 6.4.4 | 저장소 허용 목록이 분기별로 검토되며 각 항목에 대한 비즈니스 정당성 증거가 있는지 확인하십시오.       |   2   |  V   |
| 6.4.5 | 정책 위반이 아티팩트 격리 및 종속된 파이프라인 실행의 롤백을 트리거하는지 확인하십시오.            |   3   |  V   |

---

## C6.5 제3자 데이터셋 위험 평가

외부 데이터셋을 오염, 편향, 법적 준수 여부에 대해 평가하고, 전체 수명 주기 동안 이를 모니터링하십시오.

|   #   | Description                                                         | Level | Role |
| :---: | ------------------------------------------------------------------- | :---: | :--: |
| 6.5.1 | 외부 데이터셋이 오염 위험 점검(예: 데이터 지문 인식, 이상치 탐지)을 거치는지 확인하십시오.               |   1   | D/V  |
| 6.5.2 | 데이터셋 승인을 얻기 전에 편향 지표(인구 통계적 균형, 기회 균등)가 계산되었는지 확인하십시오.              |   1   |  D   |
| 6.5.3 | ML-BOM 항목에 데이터셋의 출처 및 라이선스 조건이 기록되었는지 확인하십시오.                       |   2   |  V   |
| 6.5.4 | 호스팅된 데이터셋에서 주기적인 모니터링이 변동이나 손상을 감지하는지 확인하십시오.                       |   2   |  V   |
| 6.5.5 | 자동화된 스크러빙을 통해 학습 전에 허용되지 않은 콘텐츠(저작권, 개인 식별 정보(PII))가 제거되었는지 확인하십시오. |   3   |  D   |

---

## C6.6 공급망 공격 모니터링

CVE 피드, 감사 로그 분석, 레드팀 시뮬레이션을 통해 공급망 위협을 조기에 탐지하세요.

|   #   | Description                                                         | Level | Role |
| :---: | ------------------------------------------------------------------- | :---: | :--: |
| 6.6.1 | CI/CD 감사 로그가 이상한 패키지 다운로드 또는 변조된 빌드 단계에 대한 SIEM 탐지로 스트리밍되는지 확인하십시오. |   1   |  V   |
| 6.6.2 | 사고 대응 플레이북에 손상된 모델 또는 라이브러리에 대한 롤백 절차가 포함되어 있는지 확인하십시오.             |   2   |  D   |
| 6.6.3 | 위협 인텔리전스 강화가 경고 분류 과정에서 ML 특화 지표(예: 모델 중독 IoC)를 태그하는지 확인하십시오.       |   3   |  V   |

---

## C6.7 모델 산출물용 ML-BOM

하류 소비자가 배포 시점에 구성 요소 무결성을 검증할 수 있도록 자세한 ML 전용 SBOM(ML-BOM)을 생성하고 서명합니다.

|   #   | Description                                                             | Level | Role |
| :---: | ----------------------------------------------------------------------- | :---: | :--: |
| 6.7.1 | 모든 모델 산출물이 데이터셋, 가중치, 하이퍼파라미터 및 라이선스를 나열하는 ML-BOM을 게시하는지 확인하십시오.        |   1   | D/V  |
| 6.7.2 | ML-BOM 생성과 Cosign 서명이 CI에서 자동화되어 있으며 병합을 위해 필수인지 확인하십시오.                |   1   | D/V  |
| 6.7.3 | 구성요소 메타데이터(해시, 라이선스) 중 하나라도 누락될 경우 ML-BOM 완전성 검사가 빌드를 실패로 처리하는지 확인하십시오. |   2   |  D   |
| 6.7.4 | 다운스트림 사용자가 배포 시점에 가져온 모델을 검증하기 위해 API를 통해 ML-BOM를 조회할 수 있음을 확인하십시오.     |   2   |  V   |
| 6.7.5 | ML‑BOM이 버전 관리되고 변경 사항이 비교되어 무단 수정을 감지하는지 확인하십시오.                        |   3   |  V   |
