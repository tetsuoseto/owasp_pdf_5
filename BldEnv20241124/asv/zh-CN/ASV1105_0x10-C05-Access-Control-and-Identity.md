# C5 人工智能组件与用户的访问控制与身份管理

## 控制目标

针对人工智能系统的有效访问控制需要强健的身份管理、上下文感知的授权以及遵循零信任原则的运行时执行。这些控制确保人类、服务和自主代理仅在明确授予的范围内与模型、数据和计算资源进行交互，并具备持续的验证和审计能力。

---

## C5.1 身份管理与认证

为所有实体建立基于密码学的身份，并对特权操作实施多因素认证。

|   #   | Description                                                                     | Level | Role |
| :---: | ------------------------------------------------------------------------------- | :---: | :--: |
| 5.1.1 | 验证所有人类用户和服务主体均通过集中式企业身份提供者（IdP）使用 OIDC/SAML 协议进行身份验证，并采用唯一的身份到令牌映射（不使用共享账户或凭据）。 |   1   | D/V  |
| 5.1.2 | 验证高风险操作（模型部署、权重导出、训练数据访问、生产配置更改）是否需要多因素认证或带有会话重新验证的升阶认证。                        |   1   | D/V  |
| 5.1.3 | 验证新任负责人在获得生产系统访问权限之前，是否进行了符合NIST 800-63-3 IAL-2或同等标准的身份验证。                      |   2   |  D   |
| 5.1.4 | 验证是否每季度进行访问审查，并自动检测休眠账户、强制凭证轮换以及执行注销工作流程。                                       |   2   |  V   |
| 5.1.5 | 验证联邦AI代理通过签名的JWT断言进行身份验证，该断言的最大有效期为24小时，并包含起源的加密证明。                             |   3   | D/V  |

## C5.2 资源授权与最小权限原则

为所有人工智能资源实施细粒度访问控制，采用明确的权限模型和审计追踪。

|   #   | Description                                                         | Level | Role |
| :---: | ------------------------------------------------------------------- | :---: | :--: |
| 5.2.1 | 验证所有人工智能资源（数据集、模型、端点、向量集合、嵌入索引、计算实例）是否实施基于角色的访问控制，采用明确的允许列表和默认拒绝策略。 |   1   | D/V  |
| 5.2.2 | 验证服务账户默认执行最小权限原则，从只读权限开始，并且写入权限需要有书面业务正当理由。                         |   1   | D/V  |
| 5.2.3 | 验证所有访问控制修改均与已批准的变更请求相关联，并以不可篡改的方式记录，包含时间戳、操作人员身份、资源标识符和权限差异。        |   1   |  V   |
| 5.2.4 | 验证数据分类标签（PII，PHI，出口控制，专有）是否自动传播到派生资源（嵌入、提示缓存、模型输出），并确保策略执行一致。       |   2   |  D   |
| 5.2.5 | 验证未经授权的访问尝试和权限升级事件是否在5分钟内触发带有上下文元数据的实时警报发送到SIEM系统。                  |   2   | D/V  |

---

## C5.3 动态策略评估

部署基于属性的访问控制（ABAC）引擎，实现具备审计能力的上下文感知授权决策。

|   #   | Description                                                  | Level | Role |
| :---: | ------------------------------------------------------------ | :---: | :--: |
| 5.3.1 | 验证授权决策是否外部化到专用的策略引擎（如OPA、Cedar或等效系统），并通过具备加密完整性保护的认证API进行访问。 |   1   | D/V  |
| 5.3.2 | 验证策略在运行时评估动态属性，包括用户的权限级别、资源的敏感性分类、请求上下文、租户隔离以及时间限制。          |   1   | D/V  |
| 5.3.3 | 验证策略定义在生产部署前经过版本控制、同行评审，并通过持续集成/持续部署（CI/CD）管道中的自动化测试进行验证。    |   2   |  D   |
| 5.3.4 | 验证策略评估结果包括结构化的决策理由，并传输到SIEM系统以进行关联分析和合规报告。                   |   2   |  V   |
| 5.3.5 | 验证高敏感资源的策略缓存生存时间（TTL）值不超过5分钟，且具备缓存失效功能的标准资源不超过1小时。           |   3   | D/V  |

## C5.4 查询时安全执行

实施数据库层安全控制，采用强制过滤和行级安全策略。

|   #   | Description                                                              | Level | Role |
| :---: | ------------------------------------------------------------------------ | :---: | :--: |
| 5.4.1 | 确认所有向量数据库和SQL查询都包含强制的安全过滤器（租户ID、敏感性标签、用户范围），且这些过滤器在数据库引擎级别强制执行，而非在应用代码中。 |   1   | D/V  |
| 5.4.2 | 验证所有向量数据库、搜索索引和训练数据集是否启用了行级安全（RLS）策略和字段级掩码，并支持策略继承。                      |   1   | D/V  |
| 5.4.3 | 验证失败的授权评估将通过立即中止查询并返回明确的授权错误代码，而不是返回空结果集，从而防止“混淆代理攻击”。                   |   2   |  D   |
| 5.4.4 | 验证策略评估延迟是否被持续监控，并且针对可能导致授权绕过的超时情况设置了自动警报。                                |   2   |  V   |
| 5.4.5 | 验证查询重试机制是否重新评估授权策略，以考虑活动用户会话中动态权限的变化。                                    |   3   | D/V  |

## C5.5 输出过滤与数据防泄漏

部署后处理控制以防止AI生成内容中的未经授权的数据泄露。

|   #   | Description                                              | Level | Role |
| :---: | -------------------------------------------------------- | :---: | :--: |
| 5.5.1 | 验证推理后过滤机制是否扫描并删除未经授权的个人身份信息（PII）、机密信息和专有数据，然后再将内容传递给请求者。 |   1   | D/V  |
| 5.5.2 | 验证模型输出中的引文、参考文献和来源归属是否符合调用者权限，如发现未经授权访问则予以删除。            |   1   | D/V  |
| 5.5.3 | 根据用户权限级别和数据分类，验证输出格式限制（已清理的PDF、去除元数据的图像、批准的文件类型）是否得到执行。  |   2   |  D   |
| 5.5.4 | 验证编辑算法是确定性的、版本可控的，并且保持审计日志以支持合规调查和取证分析。                  |   2   |  V   |
| 5.5.5 | 验证高风险涂黑事件是否生成自适应日志，这些日志应包含原始内容的加密哈希，以便进行取证检索且不暴露数据。      |   3   |  V   |

## C5.6 多租户隔离

确保共享人工智能基础设施中租户之间的加密和逻辑隔离。

|   #   | Description                                               | Level | Role |
| :---: | --------------------------------------------------------- | :---: | :--: |
| 5.6.1 | 验证内存空间、嵌入存储、缓存条目和临时文件是否按租户进行命名空间隔离，并在租户删除或会话终止时进行安全清除。    |   1   | D/V  |
| 5.6.2 | 验证每个 API 请求是否包含经过身份验证的租户标识符，并对其进行密码学验证，以确保其与会话上下文和用户权限一致。 |   1   | D/V  |
| 5.6.3 | 验证网络策略是否在服务网格和容器编排平台中针对跨租户通信实施默认拒绝规则。                     |   2   |  D   |
| 5.6.4 | 验证加密密钥在每个租户中是唯一的，支持客户管理密钥（CMK），并确保租户数据存储之间的加密隔离。          |   3   |  D   |

## C5.7 自主代理授权

通过作用域能力令牌和持续授权控制人工智能代理和自主系统的权限。

|   #   | Description                                                 | Level | Role |
| :---: | ----------------------------------------------------------- | :---: | :--: |
| 5.7.1 | 验证自治代理是否接收范围限定的能力令牌，该令牌明确列出允许的操作、可访问的资源、时间限制和操作约束。          |   1   | D/V  |
| 5.7.2 | 验证高风险功能（文件系统访问、代码执行、外部API调用、金融交易）默认情况下被禁用，并且激活时需要明确授权及业务理由。 |   1   | D/V  |
| 5.7.3 | 验证capability令牌是否绑定到用户会话，包含加密完整性保护，并确保它们不能在脱机场景中被持久化或重用。     |   2   |  D   |
| 5.7.4 | 验证代理发起的操作是否通过基于属性的访问控制（ABAC）策略引擎进行二次授权，确保完整的上下文评估和审计日志记录。   |   2   |  V   |
| 5.7.5 | 验证代理错误条件和异常处理是否包含能力范围信息，以支持事件分析和取证调查。                       |   3   |  V   |
