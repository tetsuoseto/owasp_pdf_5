# 9 自主编排与代理行为安全

## 控制目标

确保自主或多智能体人工智能系统只能执行明确意图的、经过身份认证的、可审计的，并且在限定的成本和风险阈值内的操作。这可以防止自主系统被攻破、工具滥用、智能体循环检测、通信劫持、身份伪装、蜂群操控和意图操控等威胁。

---

## 9.1 代理任务规划与递归预算

限制递归计划并强制在执行特权操作时设置人工检查点。

|   #   | Description                                     | Level | Role |
| :---: | ----------------------------------------------- | :---: | :--: |
| 9.1.1 | 验证每个代理执行的最大递归深度、广度、实时时钟时间、令牌数和货币成本是否集中配置并受版本控制。 |   1   | D/V  |
| 9.1.2 | 验证特权或不可逆操作（例如代码提交、财务转账）在执行前必须通过可审计渠道获得明确的人为批准。  |   1   | D/V  |
| 9.1.3 | 验证实时资源监控器在任何预算阈值被超时时触发断路器中断，从而停止进一步的任务扩展。       |   2   |  D   |
| 9.1.4 | 验证断路器事件是否已记录代理ID、触发条件和捕获的计划状态，以便进行取证审查。         |   2   | D/V  |
| 9.1.5 | 验证安全测试涵盖预算耗尽和运行计划失控场景，确认能够安全停止且无数据丢失。           |   3   |  V   |
| 9.1.6 | 确保预算策略以代码形式表达，并在持续集成/持续交付（CI/CD）中执行，以阻止配置漂移。    |   3   |  D   |

## 9.2 工具插件沙箱机制

隔离工具交互以防止未经授权的系统访问或代码执行。

|   #   | Description                                             | Level | Role |
| :---: | ------------------------------------------------------- | :---: | :--: |
| 9.2.1 | 确保每个工具/插件都在操作系统、容器或WASM级别的沙箱内执行，并采用最小权限的文件系统、网络和系统调用策略。 |   1   | D/V  |
| 9.2.2 | 验证沙箱资源配额（CPU、内存、磁盘、网络出口）和执行超时是否得到强制执行和记录。               |   1   | D/V  |
| 9.2.3 | 验证工具二进制文件或描述符是否经过数字签名；在加载之前验证签名。                        |   2   | D/V  |
| 9.2.4 | 验证沙箱的遥测数据是否流向安全信息和事件管理系统（SIEM）；异常情况（例如，尝试的出站连接）会触发警报。   |   2   |  V   |
| 9.2.5 | 确保高风险插件在投入生产部署前经过安全审核和渗透测试。                             |   3   |  V   |
| 9.2.6 | 验证沙箱逃逸尝试是否被自动阻止，并且违规插件是否被隔离等待调查。                        |   3   | D/V  |

## 9.3 自主循环与成本界定

检测并阻止无控制的代理间递归及成本爆炸。

|   #   | Description                          | Level | Role |
| :---: | ------------------------------------ | :---: | :--: |
| 9.3.1 | 验证代理间调用是否包含跳数限制或生存时间（TTL），由运行时递减并执行。 |   1   | D/V  |
| 9.3.2 | 验证代理是否保持唯一的调用图ID，以识别自我调用或循环模式。       |   2   |  D   |
| 9.3.3 | 验证累积计算单元和支出计数器是否按请求链进行跟踪；超出限制将中断该链。  |   2   | D/V  |
| 9.3.4 | 验证形式分析或模型检验能够证明代理协议中不存在无界递归。         |   3   |  V   |
| 9.3.5 | 验证循环中止事件是否生成警报并提供持续改进指标。             |   3   |  D   |

## 9.4 协议级误用保护

在代理和外部系统之间建立安全通信通道，以防止劫持或篡改。

|   #   | Description                                          | Level | Role |
| :---: | ---------------------------------------------------- | :---: | :--: |
| 9.4.1 | 验证所有代理到工具和代理到代理的消息是否经过身份验证（例如，双向 TLS 或 JWT）并实现端到端加密。 |   1   | D/V  |
| 9.4.2 | 验证模式是否被严格校验；未知字段或格式错误的消息应被拒绝。                        |   1   |  D   |
| 9.4.3 | 验证完整性检查（MAC 或数字签名）是否覆盖整个消息负载，包括工具参数。                 |   2   | D/V  |
| 9.4.4 | 验证在协议层强制执行了重放保护（随机数或时间戳窗口）。                          |   2   |  D   |
| 9.4.5 | 验证协议实现是否进行了模糊测试和静态分析，以检测注入或反序列化漏洞。                   |   3   |  V   |

## 9.5 代理身份与防篡改证据

确保操作可追溯，修改可检测。

|   #   | Description                     | Level | Role |
| :---: | ------------------------------- | :---: | :--: |
| 9.5.1 | 验证每个代理实例是否拥有唯一的加密身份（密钥对或硬件根凭证）。 |   1   | D/V  |
| 9.5.2 | 验证所有代理操作均已签名和时间戳；日志包含签名以防止否认。   |   2   | D/V  |
| 9.5.3 | 验证防篡改日志是否存储在仅追加或一次性写入的介质中。      |   2   |  V   |
| 9.5.4 | 验证身份密钥是否按照定义的时间表以及在发生泄露指标时进行轮换。 |   3   |  D   |
| 9.5.5 | 验证欺骗或密钥冲突尝试是否会立即触发受影响代理的隔离。     |   3   | D/V  |

## 9.6 多智能体群体风险降低

通过隔离和正式安全建模来减轻群体行为风险。

|   #   | Description                             | Level | Role |
| :---: | --------------------------------------- | :---: | :--: |
| 9.6.1 | 验证在不同安全域中运行的代理是否在隔离的运行时沙箱或网络分段中执行。      |   1   | D/V  |
| 9.6.2 | 在部署之前，验证群体行为是否已被建模并正式验证其活性和安全性。         |   3   |  V   |
| 9.6.3 | 验证运行时监控器能检测到新出现的不安全模式（例如，振荡、死锁）并启动纠正措施。 |   3   |  D   |

## 9.7 用户与工具认证/授权

为每个代理触发的操作实施强有力的访问控制。

|   #   | Description                            | Level | Role |
| :---: | -------------------------------------- | :---: | :--: |
| 9.7.1 | 验证代理作为一级主体对下游系统进行身份验证，且绝不重用终端用户凭据。     |   1   | D/V  |
| 9.7.2 | 验证细粒度授权策略是否限制代理可以调用的工具以及其可以提供的参数。      |   2   |  D   |
| 9.7.3 | 验证权限检查是否在每次调用时重新评估（持续授权），而不仅仅在会话开始时进行。 |   2   |  V   |
| 9.7.4 | 验证委托权限是否会自动过期，并在超时或权限范围更改后要求重新同意。      |   3   |  D   |

## 9.8 代理间通信安全

加密并完整性保护所有代理间消息，以防止窃听和篡改。

|   #   | Description                          | Level | Role |
| :---: | ------------------------------------ | :---: | :--: |
| 9.8.1 | 验证代理通道必须实现相互认证和完美前向保密加密（例如 TLS 1.3）。 |   1   | D/V  |
| 9.8.2 | 验证消息完整性和来源在处理前是否经过确认；失败时发出警报并丢弃该消息。  |   1   |  D   |
| 9.8.3 | 验证通信元数据（时间戳、序列号）是否被记录，以支持取证重建。       |   2   | D/V  |
| 9.8.4 | 验证形式验证或模型检测是否确认协议状态机无法进入不安全状态。       |   3   |  V   |

## 9.9 意图验证与约束执行

验证代理操作是否符合用户的陈述意图和系统约束。

|   #   | Description                                   | Level | Role |
| :---: | --------------------------------------------- | :---: | :--: |
| 9.9.1 | 验证预执行约束求解器是否根据硬编码的安全和政策规则检查建议的行动。             |   1   |  D   |
| 9.9.2 | 验证高影响操作（财务、破坏性、隐私敏感）是否需要启动用户的明确意图确认。          |   2   | D/V  |
| 9.9.3 | 验证后置条件检查，以确认已完成的操作实现了预期效果且无副作用；若发现差异，则触发回滚。   |   2   |  V   |
| 9.9.4 | 验证形式方法（例如，模型检查、定理证明）或基于属性的测试能证明代理计划满足所有声明的约束。 |   3   |  V   |
| 9.9.5 | 验证意图不匹配或约束违规事件是否推动持续改进循环和威胁情报共享。              |   3   |  D   |

## 9.10 代理推理策略安全

安全选择和执行包括ReAct、Chain-of-Thought和Tree-of-Thoughts方法在内的不同推理策略。

|   #    | Description                                                                      | Level | Role |
| :----: | -------------------------------------------------------------------------------- | :---: | :--: |
| 9.10.1 | 验证推理策略选择使用确定性标准（输入复杂度、任务类型、安全上下文），并且在相同安全上下文中，相同输入产生相同的策略选择。                     |   1   | D/V  |
| 9.10.2 | 验证每种推理策略（ReAct、Chain-of-Thought、Tree-of-Thoughts）是否具有针对其认知方法的专用输入验证、输出清理和执行时间限制。 |   1   | D/V  |
| 9.10.3 | 验证推理策略转换是否已记录完整上下文，包括输入特征、选择标准值和执行元数据，以便审计轨迹重建。                                  |   2   | D/V  |
| 9.10.4 | 验证思维树推理是否包括分支剪枝机制，当检测到策略违规、资源限制或安全边界时终止探索。                                       |   2   | D/V  |
| 9.10.5 | 验证ReAct（推理-行动-观察）循环在每个阶段包含验证检查点：推理步骤验证、行动授权以及在继续之前的观察数据清理。                       |   2   | D/V  |
| 9.10.6 | 验证推理策略的性能指标（执行时间、资源使用、输出质量）是否在指标超出配置阈值时通过自动警报进行监控。                               |   3   | D/V  |
| 9.10.7 | 验证结合多种策略的混合推理方法是否保持所有组成策略的输入验证和输出约束，且不绕过任何安全控制。                                  |   3   | D/V  |
| 9.10.8 | 验证推理策略安全测试包括使用格式错误的输入进行模糊测试、设计用于强制策略切换的对抗性提示，以及对每种认知方法进行边界条件测试。                  |   3   | D/V  |

## 9.11 代理生命周期状态管理与安全

通过加密审计跟踪和定义的恢复程序，实现安全代理的初始化、状态转换和终止。

|   #    | Description                                                                | Level | Role |
| :----: | -------------------------------------------------------------------------- | :---: | :--: |
| 9.11.1 | 验证代理初始化是否包括使用硬件支持的凭证建立加密身份，以及包含代理ID、时间戳、配置哈希和初始化参数的不可变启动审计日志。              |   1   | D/V  |
| 9.11.2 | 验证代理状态转换是否经过加密签名、时间戳标注，并完整记录上下文信息，包括触发事件、前一状态哈希、新状态哈希以及执行的安全验证。            |   2   | D/V  |
| 9.11.3 | 验证代理关闭程序是否包括使用加密擦除或多次覆盖进行安全内存清理，凭证撤销并通知证书授权机构，以及生成防篡改的终止证书。                |   2   | D/V  |
| 9.11.4 | 验证代理恢复机制使用加密校验和（至少 SHA-256）来验证状态完整性，并在检测到损坏时回滚到已知良好状态，同时配备自动警报和人工批准要求。     |   3   | D/V  |
| 9.11.5 | 验证代理持久化机制是否使用每个代理的 AES-256 密钥加密敏感状态数据，并在可配置的时间表（最长90天）上实现安全的密钥轮换，且支持零停机部署。 |   3   | D/V  |

## 9.12 工具集成安全框架

动态工具加载、执行和结果验证的安全控制，包含定义的风险评估和审批流程。

|   #    | Description                                                                | Level | Role |
| :----: | -------------------------------------------------------------------------- | :---: | :--: |
| 9.12.1 | 验证工具描述符是否包含安全元数据，指定所需权限（读/写/执行）、风险级别（低/中/高）、资源限制（CPU、内存、网络）以及工具清单中记录的验证要求。 |   1   | D/V  |
| 9.12.2 | 验证工具执行结果是否符合预期的模式（JSON模式、XML模式）和安全策略（输出清理、数据分类），并在集成前应用超时限制和错误处理程序。        |   1   | D/V  |
| 9.12.3 | 验证工具交互日志是否包含详细的安全上下文，包括权限使用情况、数据访问模式、执行时间、资源消耗和返回代码，并采用结构化日志以便于SIEM集成。     |   2   | D/V  |
| 9.12.4 | 验证动态工具加载机制是否使用公钥基础设施（PKI）验证数字签名，并在执行前实施具有沙箱隔离和权限验证的安全加载协议。                 |   2   | D/V  |
| 9.12.5 | 验证工具安全评估是否针对新版本自动触发，且包含强制审批关卡，包括静态分析、动态测试和安全团队审查，同时具备书面审批标准和服务水平协议（SLA）要求。 |   3   | D/V  |
