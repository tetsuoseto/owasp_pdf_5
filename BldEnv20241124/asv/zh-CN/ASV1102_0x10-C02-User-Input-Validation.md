# C2 用户输入验证

## 控制目标

对用户输入的强健验证是防御针对人工智能系统的一些最具破坏性攻击的第一道防线。提示注入攻击可以覆盖系统指令、泄露敏感数据，或引导模型产生不被允许的行为。研究表明，除非设置专门的过滤器和指令层级，否则利用非常长上下文窗口的“多次注入”越狱攻击将会有效。此外，细微的对抗扰动攻击，例如同形字替换或“1337”语言（leet speak）交换，能够悄无声息地改变模型的决策。

---

## C2.1 提示注入防御

提示注入是人工智能系统面临的主要风险之一。对此类攻击的防御措施结合了静态模式过滤器、动态分类器以及指令层级执行等多种方法。

|   #   | Description                                                             | Level | Role |
| :---: | ----------------------------------------------------------------------- | :---: | :--: |
| 2.1.1 | 验证用户输入是否经过筛查，确保其符合不断更新的已知提示注入模式库（包括越狱关键词、“忽略之前内容”、角色扮演链条、间接HTML/URL攻击）。 |   1   | D/V  |
| 2.1.2 | 验证系统是否强制执行指令层级，其中系统或开发者消息优先于用户指令，即使在上下文窗口扩展后也是如此。                       |   1   | D/V  |
| 2.1.3 | 验证对抗性评估测试（例如，红队“多次示例”提示）在每次模型或提示模板发布前均已执行，并设有成功率阈值和用于防止性能回退的自动阻断机制。     |   2   | D/V  |
| 2.1.4 | 验证来自第三方内容（网页、PDF、电子邮件）的提示是否在被连接到主提示之前，在隔离的解析环境中进行了清理。                   |   2   |  D   |
| 2.1.5 | 验证所有提示过滤规则更新、分类器模型版本和黑名单更改均受版本控制且可审计。                                   |   3   | D/V  |

---

## C2.2 对抗样本抵抗能力

自然语言处理（NLP）模型仍然容易受到人类常忽视但模型容易误分类的细微字符或单词级扰动的影响。

|   #   | Description                                       | Level | Role |
| :---: | ------------------------------------------------- | :---: | :--: |
| 2.2.1 | 验证基本输入归一化步骤（Unicode NFC，同形异义字映射，空白字符修剪）是否在分词之前执行。 |   1   |  D   |
| 2.2.2 | 验证统计异常检测是否能够标记与语言规范有异常高编辑距离、过多重复标记或异常嵌入距离的输入。     |   2   | D/V  |
| 2.2.3 | 验证推理管道是否支持可选的对抗训练强化模型变体或防御层（例如随机化、防御蒸馏），以保护高风险端点。 |   2   |  D   |
| 2.2.4 | 验证可疑的对抗性输入是否被隔离，并在去除个人识别信息后完整记录其负载内容。             |   2   |  V   |
| 2.2.5 | 验证鲁棒性指标（已知攻击套件的成功率）是否随着时间跟踪，并且回归会触发发布阻塞。          |   3   | D/V  |

---

## C2.3 模式、类型和长度验证

包含格式错误或超大输入的AI攻击可能导致解析错误、提示内容跨字段溢出以及资源耗尽。严格的模式（schema）强制执行也是进行确定性工具调用的前提条件。

|   #   | Description                                                                   | Level | Role |
| :---: | ----------------------------------------------------------------------------- | :---: | :--: |
| 2.3.1 | 验证每个 API 或函数调用端点是否定义了明确的输入模式（JSON Schema、Protobuf 或多模态等效格式），并且在组装提示前对输入进行了验证。 |   1   |  D   |
| 2.3.2 | 验证超过最大令牌或字节限制的输入是否被安全错误拒绝，且绝不被静默截断。                                           |   1   | D/V  |
| 2.3.3 | 验证类型检查（例如，数值范围、枚举值、图像/音频的MIME类型）是在服务器端强制执行的，而不仅仅是在客户端代码中。                     |   2   | D/V  |
| 2.3.4 | 验证语义验证器（例如 JSON Schema）以恒定时间运行，以防止算法拒绝服务攻击。                                   |   2   |  D   |
| 2.3.5 | 验证失败时，确保日志记录经过编辑的有效载荷片段和明确的错误代码，以协助安全分类。                                      |   3   |  V   |

---

## C2.4 内容与政策筛查

开发者应该能够检测请求禁止内容（如非法指令、仇恨言论和受版权保护文本）的语法上有效的提示，然后防止其传播。

|   #   | Description                                             | Level | Role |
| :---: | ------------------------------------------------------- | :---: | :--: |
| 2.4.1 | 验证内容分类器（零样本或微调）是否对每个输入评分，包括暴力、自残、仇恨、色情内容和非法请求，并支持可配置阈值。 |   1   |  D   |
| 2.4.2 | 验证违反政策的输入将收到标准化的拒绝或安全完成，以确保它们不会传播到下游的大型语言模型调用。          |   1   | D/V  |
| 2.4.3 | 验证筛查模型或规则集至少每季度重新训练/更新一次，纳入新观察到的越狱或政策规避模式。              |   2   |  D   |
| 2.4.4 | 通过在请求时解析基于属性的规则，验证筛选是否遵守用户特定的政策（年龄、地区法律限制）。             |   2   |  D   |
| 2.4.5 | 验证筛查日志是否包含分类器置信度分数和策略类别标签，以便进行安全运营中心（SOC）关联和未来的红队重放。    |   3   |  V   |

---

## C2.5 输入速率限制与滥用防护

开发人员应通过限制输入速率和检测异常使用模式来防止对人工智能系统的滥用、资源耗尽和自动化攻击。

|   #   | Description                               | Level | Role |
| :---: | ----------------------------------------- | :---: | :--: |
| 2.5.1 | 验证所有输入端点是否对每个用户、每个IP和每个API密钥的速率限制进行了强制执行。 |   1   | D/V  |
| 2.5.2 | 验证突发和持续速率限制是否已调整，以防止拒绝服务（DoS）和暴力破解攻击。     |   2   | D/V  |
| 2.5.3 | 验证异常使用模式（例如，快速连续请求、输入泛滥）是否会触发自动阻断或升级处理。   |   2   | D/V  |
| 2.5.4 | 验证滥用防范日志是否被保留并用于审查新出现的攻击模式。               |   3   |  V   |

---

## C2.6 多模态输入验证

人工智能系统应包含对非文本输入（图像、音频、文件）的强健验证，以防止注入、规避或资源滥用。

|   #   | Description                         | Level | Role |
| :---: | ----------------------------------- | :---: | :--: |
| 2.6.1 | 在处理之前，验证所有非文本输入（图像、音频、文件）的类型、大小和格式。 |   1   |  D   |
| 2.6.2 | 确认在导入之前，文件已被扫描以检测恶意软件和隐写负载。         |   2   | D/V  |
| 2.6.3 | 验证图像/音频输入是否被检查过，确保没有敌意扰动或已知攻击模式。    |   2   | D/V  |
| 2.6.4 | 验证多模态输入验证失败是否被记录并触发警报以供调查。          |   3   |  V   |

---

## C2.7 输入来源与归属

人工智能系统应通过监控和标记所有用户输入的来源，支持审计、滥用跟踪和合规性。

|   #   | Description                              | Level | Role |
| :---: | ---------------------------------------- | :---: | :--: |
| 2.7.1 | 验证所有用户输入在接收时均附带元数据（用户ID、会话、来源、时间戳、IP地址）。 |   1   | D/V  |
| 2.7.2 | 验证所有处理过的输入的来源元数据是否被保留且可审计。               |   2   | D/V  |
| 2.7.3 | 验证异常或不可信的输入来源是否被标记，并且受到加强的审查或阻止。         |   2   | D/V  |

---

## C2.8 实时自适应威胁检测

开发者应采用先进的AI威胁检测系统，该系统能够适应新的攻击模式，并通过编译的模式匹配提供实时保护。

|   #   | Description                                       | Level | Role |
| :---: | ------------------------------------------------- | :---: | :--: |
| 2.8.1 | 验证威胁检测模式是否已编译为优化的正则表达式引擎，以实现高性能的实时过滤并最大限度地减少延迟影响。 |   1   | D/V  |
| 2.8.2 | 验证威胁检测系统是否为不同的威胁类别（提示注入、有害内容、敏感数据、系统命令）维护独立的模式库。  |   1   | D/V  |
| 2.8.3 | 验证自适应威胁检测是否包含根据攻击频率和成功率更新威胁敏感性的机器学习模型。            |   2   | D/V  |
| 2.8.4 | 验证实时威胁情报源是否自动使用新的攻击特征和妥协指标（IOC）更新模式库。             |   2   | D/V  |
| 2.8.5 | 验证威胁检测的误报率是否被持续监控，并自动调整模式特异性以最小化对合法使用场景的干扰。       |   3   | D/V  |
| 2.8.6 | 验证上下文威胁分析是否考虑输入来源、用户行为模式和会话历史，以提高检测准确性。           |   3   | D/V  |
| 2.8.7 | 验证威胁检测性能指标（检测率、处理延迟、资源利用率）是否实时监控和优化。              |   3   | D/V  |

## C2.9 多模态安全验证流程

开发人员应针对文本、图像、音频及其他人工智能输入方式，提供安全验证，包括特定类型的威胁检测和资源隔离。

|   #   | Description                                                       | Level | Role |
| :---: | ----------------------------------------------------------------- | :---: | :--: |
| 2.9.1 | 验证每种输入模态是否具有专门的安全验证器，并附有记录的威胁模式（文本：提示注入，图像：隐写术，音频：频谱图攻击）和检测阈值。    |   1   | D/V  |
| 2.9.2 | 验证多模态输入是否在隔离的沙箱中处理，并为每种模态类型定义了特定的资源限制（内存、CPU、处理时间），并在安全策略中进行了记录。  |   2   | D/V  |
| 2.9.3 | 验证跨模态攻击检测是否能够通过关联规则和告警生成识别涵盖多种输入类型的协调攻击（例如，图像中的隐写载荷与文本中的提示注入相结合）。 |   2   | D/V  |
| 2.9.4 | 验证多模态验证失败是否会触发详细日志记录，包括所有输入模态、验证结果、威胁评分以及与结构化日志格式用于SIEM集成的关联分析。   |   3   | D/V  |
| 2.9.5 | 验证特定模态内容分类器是否根据记录的计划（至少每季度一次）更新，内容包括新的威胁模式、对抗样本，并确保性能基准维持在基线阈值以上。 |   3   | D/V  |
