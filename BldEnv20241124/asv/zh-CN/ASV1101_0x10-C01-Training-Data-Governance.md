# C1 训练数据治理与偏差管理

## 控制目标

训练数据必须以保护来源、安全性、质量和公平性的方式进行采集、处理和维护。这样做既履行了法律责任，也减少了在整个人工智能生命周期中出现偏见、投毒或隐私泄露的风险。

---

## C1.1 训练数据来源

保持所有数据集的可验证清单，仅接受可信来源，并记录所有变更以确保可审计性。

|   #   | Description                                                      | Level | Role |
| :---: | ---------------------------------------------------------------- | :---: | :--: |
| 1.1.1 | 确保维护每个训练数据源的最新清单（包括来源、管理员/所有者、许可、收集方法、预期使用限制和处理历史）。              |   1   | D/V  |
| 1.1.2 | 确保仅允许经过质量、代表性、伦理来源和许可合规性审核的数据集，降低中毒、内嵌偏见和知识产权侵权的风险。              |   1   | D/V  |
| 1.1.3 | 验证是否执行了数据最小化，以排除不必要或敏感的属性。                                       |   1   | D/V  |
| 1.1.4 | 确保所有数据集的更改都必须经过记录的审批流程。                                          |   2   | D/V  |
| 1.1.5 | 通过审阅者的交叉检查或共识，确保标签/标注质量。                                         |   2   | D/V  |
| 1.1.6 | 确认为重要的训练数据集维护“数据卡”或“数据集数据表”，详细说明其特征、动机、组成、收集过程、预处理以及推荐/不推荐的使用方法。 |   2   | D/V  |

---

## C1.2 训练数据安全性与完整性

限制访问，对静态数据和传输中的数据进行加密，并验证完整性以防止篡改或盗窃。

|   #   | Description                                                             | Level | Role |
| :---: | ----------------------------------------------------------------------- | :---: | :--: |
| 1.2.1 | 验证存储和数据管道受到访问控制的保护。                                                     |   1   | D/V  |
| 1.2.2 | 验证数据集在传输过程中是否加密，并且对于所有敏感或个人身份信息（PII），在静止状态下是否使用行业标准的加密算法和密钥管理实践进行加密。    |   2   | D/V  |
| 1.2.3 | 验证在存储和传输过程中使用加密哈希或数字签名以确保数据完整性，并应用自动异常检测技术以防止未经授权的修改或损坏，包括针对性的数据信息投毒尝试。 |   2   | D/V  |
| 1.2.4 | 验证数据集版本是否被跟踪以支持回滚。                                                      |   3   | D/V  |
| 1.2.5 | 核实过时数据是否按照数据保留政策、监管要求安全清除或匿名化，以缩小攻击面。                                   |   2   | D/V  |

---

## C1.3 表示的完整性与公平性

检测人口统计偏差并采取缓解措施，以确保模型在各组之间的错误率公平。

|   #   | Description                                                                                | Level | Role |
| :---: | ------------------------------------------------------------------------------------------ | :---: | :--: |
| 1.3.1 | 验证数据集是否针对具有法律保护的属性（例如种族、性别、年龄）以及与模型应用领域相关的其他伦理敏感特征（例如社会经济状况、地理位置）进行了代表性不平衡和潜在偏差的分析。        |   1   | D/V  |
| 1.3.2 | 验证已识别的偏差是否通过文档化的策略得到缓解，例如重新平衡、有针对性的数据增强、算法调整（例如预处理、中间处理、后处理技术）或重新加权，并评估缓解措施对公平性和整体模型性能的影响。 |   2   | D/V  |
| 1.3.3 | 验证已评估并记录训练后公平性指标。                                                                          |   2   | D/V  |
| 1.3.4 | 验证生命周期偏差管理策略是否分配了责任人和审核周期。                                                                 |   3   | D/V  |

---

## C1.4 训练数据标注的质量、完整性与安全性

通过加密保护标签，并对关键类别要求双重审核。

|   #   | Description                                                            | Level | Role |
| :---: | ---------------------------------------------------------------------- | :---: | :--: |
| 1.4.1 | 确保通过明确的指导方针、复审员交叉检查、一致性机制（例如，监控标注者间一致性），以及定义的差异解决流程，来保证标注/注释质量。        |   2   | D/V  |
| 1.4.2 | 验证是否对标签工件应用了加密哈希或数字签名，以确保其完整性和真实性。                                     |   2   | D/V  |
| 1.4.3 | 验证标注接口和平台是否实施强访问控制，维护所有标注活动的防篡改审计日志，并防止未经授权的修改。                        |   2   | D/V  |
| 1.4.4 | 验证对安全性、保密性或公平性至关重要的标签（例如，识别有害内容、关键医疗发现）是否接受了强制性的独立双重审核或等效的严格验证。        |   3   | D/V  |
| 1.4.5 | 验证是否根据数据最小化原则，对标签中无意捕获或必然存在的敏感信息（包括个人身份信息）在静态和传输过程中进行了脱敏、假名化、匿名化或加密处理。 |   2   | D/V  |
| 1.4.6 | 确保标签指南和说明全面，具有版本控制，并经过同行评审。                                            |   2   | D/V  |
| 1.4.7 | 确保数据架构（包括标签的架构）定义清晰，并进行版本控制。                                           |   2   | D/V  |
| 1.8.2 | 验证外包或众包标注工作流程中是否包含技术/程序保障措施，以确保数据的机密性、完整性、标签质量，并防止数据泄露。                |   2   | D/V  |

---

## C1.5 训练数据质量保证

结合自动验证、人工抽查和记录的补救措施，以确保数据集的可靠性。

|   #   | Description                                                                  | Level | Role |
| :---: | ---------------------------------------------------------------------------- | :---: | :--: |
| 1.5.1 | 验证自动化测试是否能够捕捉每次摄取或重大转换中的格式错误、空值和标签偏差。                                        |   1   |  D   |
| 1.5.2 | 验证失败的数据集是否被隔离并有审计跟踪。                                                         |   1   | D/V  |
| 1.5.3 | 验证领域专家的手动抽查覆盖了具有统计显著性的样本（例如，≥1%或1000个样本，以较大者为准，或根据风险评估确定），以识别自动化未检测出的细微质量问题。 |   2   |  V   |
| 1.5.4 | 验证补救步骤是否已附加到溯源记录中。                                                           |   2   | D/V  |
| 1.5.5 | 确保质量门禁阻止不合格的数据集，除非获得例外批准。                                                    |   2   | D/V  |

---

## C1.6 数据投毒检测

应用统计异常检测和隔离工作流程以阻止对抗性插入。

|   #   | Description                                                               | Level | Role |
| :---: | ------------------------------------------------------------------------- | :---: | :--: |
| 1.6.1 | 验证异常检测技术（例如，统计方法、异常值检测、嵌入分析）是否在数据摄取阶段及主要训练运行之前应用于训练数据，以识别潜在的投毒攻击或无意的数据损坏。 |   2   | D/V  |
| 1.6.2 | 验证被标记的样本在训练前会触发人工审核。                                                      |   2   | D/V  |
| 1.6.3 | 验证结果是否反馈到模型的安全档案中，并为持续的威胁情报提供信息。                                          |   2   |  V   |
| 1.6.4 | 验证检测逻辑是否已使用新的威胁情报进行了刷新。                                                   |   3   | D/V  |
| 1.6.5 | 验证在线学习管道是否监控分布漂移。                                                         |   3   | D/V  |
| 1.6.6 | 验证是否根据系统的风险状况和数据来源，考虑并实施了针对已知数据投毒攻击类型（例如标签翻转、后门触发器插入、影响实例攻击）的特定防御措施。      |   3   | D/V  |

---

## C1.7 用户数据删除与同意执行

尊重所有数据集、备份和派生产物中的删除和撤回同意请求。

|   #   | Description                                                              | Level | Role |
| :---: | ------------------------------------------------------------------------ | :---: | :--: |
| 1.7.1 | 验证删除工作流程是否清除主数据和派生数据，并评估对模型的影响；同时评估受影响模型的影响，并在必要时进行处理（例如，通过重新训练或重新校准）。   |   1   | D/V  |
| 1.7.2 | 验证已建立机制以跟踪和遵守用户对用于训练数据的同意范围和状态（包括撤回），并且在将数据纳入新的训练过程或重要模型更新之前，必须确认同意的有效性。 |   2   |  D   |
| 1.7.3 | 验证工作流程是否每年进行测试并记录。                                                       |   2   |  V   |

---

## C1.8 供应链安全

审查外部数据提供商，并通过认证的加密通道验证数据完整性。

|   #   | Description                                                                  | Level | Role |
| :---: | ---------------------------------------------------------------------------- | :---: | :--: |
| 1.8.1 | 确保第三方数据供应商，包括预训练模型提供者和外部数据集提供者，在其数据或模型被集成之前，经过安全、隐私、伦理来源和数据质量的尽职调查。          |   2   | D/V  |
| 1.8.2 | 验证外部传输是否使用TLS/身份验证和完整性检查。                                                    |   1   |  D   |
| 1.8.3 | 验证高风险数据源（例如，来源不明的开源数据集、未经审核的供应商）在用于敏感应用之前，是否经过加强审查，如沙盒分析、广泛的质量/偏差检查及针对性中毒检测。 |   2   | D/V  |
| 1.8.4 | 在微调或部署之前，确认从第三方获得的预训练模型已进行嵌入偏差、潜在后门、架构完整性及其原始训练数据来源的评估。                      |   3   | D/V  |

---

## C1.9 对抗样本检测

在训练阶段实施措施，例如对抗训练，以增强模型对对抗样本的抵抗力。

|   #   | Description                                                       | Level | Role |
| :---: | ----------------------------------------------------------------- | :---: | :--: |
| 1.9.1 | 根据风险评估，验证是否为相关模型实施并调优了适当的防御措施，如对抗训练（使用生成的对抗样本）、带扰动输入的数据增强或鲁棒优化技术。 |   3   | D/V  |
| 1.9.2 | 验证如使用对抗性训练，是否对抗性数据集的生成、管理和版本控制都有记录和控制。                            |   2   | D/V  |
| 1.9.3 | 验证对抗鲁棒性训练对模型性能（针对干净和对抗输入）及公平性指标的影响是否进行了评估、记录和监控。                  |   3   | D/V  |
| 1.9.4 | 确保针对对抗训练和鲁棒性的策略定期进行审查和更新，以应对不断演变的对抗攻击技术。                          |   3   | D/V  |

---

## C1.10 数据血缘和可追溯性

跟踪每个数据点从来源到模型输入的完整过程，以便审计和事件响应。

|   #    | Description                            | Level | Role |
| :----: | -------------------------------------- | :---: | :--: |
| 1.10.1 | 验证每个数据点的血统，包括所有的转换、增强和合并，是否已被记录并且可以重建。 |   2   | D/V  |
| 1.10.2 | 验证血统记录是不可更改的，安全存储的，并且可以用于审计或调查。        |   2   | D/V  |
| 1.10.3 | 验证血统追踪是否涵盖原始数据和合成数据。                   |   2   | D/V  |

---

## C1.11 合成数据管理

确保合成数据得到妥善管理、标注和风险评估。

|   #    | Description                          | Level | Role |
| :----: | ------------------------------------ | :---: | :--: |
| 1.11.1 | 确保在整个流程中所有合成数据均被明确标注，且可与真实数据区分开来。    |   2   | D/V  |
| 1.11.2 | 验证合成数据的生成过程、参数和预期用途是否有文档记录。          |   2   | D/V  |
| 1.11.3 | 在用于训练之前，确保对合成数据进行偏差、隐私泄露和表现性问题的风险评估。 |   2   | D/V  |

---

## C1.12 数据访问监控与异常检测

监控并警报对训练数据的异常访问，以检测内部威胁或数据泄露。

|   #    | Description                        | Level | Role |
| :----: | ---------------------------------- | :---: | :--: |
| 1.12.1 | 验证所有对训练数据的访问都被记录，包括用户、时间和操作。       |   2   | D/V  |
| 1.12.2 | 验证访问日志是否定期审查异常模式，例如大规模导出或来自新地点的访问。 |   2   | D/V  |
| 1.12.3 | 验证可疑访问事件是否生成了警报并及时进行调查。            |   2   | D/V  |

---

## C1.13 数据保留与过期政策

定义并执行数据保留期限，以尽量减少不必要的数据存储。

|   #    | Description                     | Level | Role |
| :----: | ------------------------------- | :---: | :--: |
| 1.13.1 | 验证所有训练数据集是否定义了明确的保留期限。          |   1   | D/V  |
| 1.13.2 | 验证数据集在其生命周期结束时是否自动到期、删除或进行删除审核。 |   2   | D/V  |
| 1.13.3 | 验证保留和删除操作是否被记录和可审计。             |   2   | D/V  |

---

## C1.14 监管与司法合规

确保所有训练数据符合适用的法律法规。

|   #    | Description                              | Level | Role |
| :----: | ---------------------------------------- | :---: | :--: |
| 1.14.1 | 验证所有数据集是否已识别并执行数据驻留和跨境传输要求。              |   2   | D/V  |
| 1.14.2 | 验证行业特定法规（例如，医疗保健、金融）是否已被识别并在数据处理过程中得到遵守。 |   2   | D/V  |
| 1.14.3 | 确认相关隐私法律（例如 GDPR、CCPA）的合规性已被记录并定期审查。     |   2   | D/V  |

---

## C1.15 数据水印与指纹识别

检测未经授权的专有或敏感数据的重用或泄露。

|   #    | Description                 | Level | Role |
| :----: | --------------------------- | :---: | :--: |
| 1.15.1 | 确保在可行的情况下，对数据集或子集进行水印或指纹标记。 |   3   | D/V  |
| 1.15.2 | 验证水印/指纹识别方法不会引入偏见或隐私风险。     |   3   | D/V  |
| 1.15.3 | 验证是否定期进行检查以检测水印数据的未授权重用或泄露。 |   3   | D/V  |

## C1.16 数据主体权利管理

支持数据主体权利，如访问、更正、限制和反对。

|   #    | Description                   | Level | Role |
| :----: | ----------------------------- | :---: | :--: |
| 1.16.1 | 验证是否存在响应数据主体请求访问、纠正、限制或反对的机制。 |   2   | D/V  |
| 1.16.2 | 验证请求是否在法律规定的时间范围内被记录、跟踪和完成。   |   2   | D/V  |
| 1.16.3 | 验证数据主体权利流程是否定期进行测试和审查以确保其有效性。 |   2   | D/V  |

## C1.17 数据集版本影响分析

在更新或替换版本之前，评估数据集变化的影响。

|   #    | Description                            | Level | Role |
| :----: | -------------------------------------- | :---: | :--: |
| 1.17.1 | 在更新或替换数据集版本之前，确保进行影响分析，涵盖模型性能、公平性和合规性。 |   2   | D/V  |
| 1.17.2 | 验证影响分析的结果已被相关利益相关者记录并审查。               |   2   | D/V  |
| 1.17.3 | 验证是否存在回滚计划，以防新版本带来不可接受的风险或回退问题。        |   2   | D/V  |

确保参与数据标注的人员的安全和完整性。

|   #    | Description                           | Level | Role |
| :----: | ------------------------------------- | :---: | :--: |
| 1.18.1 | 确认所有参与数据标注的人员都经过背景调查并接受过数据安全和隐私方面的培训。 |   2   | D/V  |
| 1.18.2 | 确保所有标注人员签署保密和不披露协议。                   |   2   | D/V  |
| 1.18.3 | 验证注释平台是否实施访问控制并监控内部威胁。                |   2   | D/V  |
