# โครงสร้างพื้นฐาน C4, การกำหนดค่า และความมั่นคงปลอดภัยในการปรับใช้

## วัตถุประสงค์การควบคุม

โครงสร้างพื้นฐานของปัญญาประดิษฐ์จะต้องแข็งแกร่งต่อการยกระดับสิทธิ์, การปรับเปลี่ยนห่วงโซ่อุปทาน, และการเคลื่อนที่ในแนวข้างผ่านการตั้งค่าที่ปลอดภัย, การแยกการทำงานระหว่างรันไทม์, ท่อการปรับใช้ที่เชื่อถือได้, และการตรวจสอบอย่างครอบคลุม ส่วนประกอบและการตั้งค่าของโครงสร้างพื้นฐานที่ได้รับอนุญาตและตรวจสอบแล้วเท่านั้นที่จะเข้าสู่การใช้งานจริงผ่านกระบวนการที่ควบคุมไว้ซึ่งรักษาความปลอดภัย, ความสมบูรณ์, และความสามารถในการตรวจสอบได้

เป้าหมายความปลอดภัยหลัก: องค์ประกอบโครงสร้างพื้นฐานที่ลงนามด้วยระบบเข้ารหัสและสแกนหาช่องโหว่เท่านั้นที่จะถึงขั้นตอนการผลิตผ่านระบบตรวจสอบอัตโนมัติที่บังคับใช้มาตรการรักษาความปลอดภัยและรักษาบันทึกตรวจสอบที่ไม่สามารถเปลี่ยนแปลงได้

## C4.1 การแยกสภาพแวดล้อมรันไทม์

ป้องกันการหลบหนีจากคอนเทนเนอร์และการยกระดับสิทธิ์ผ่านหน่วยแยกชั้นเคอร์เนลและการควบคุมการเข้าถึงแบบบังคับ

| :---: | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 4.1.1 | ตรวจสอบให้แน่ใจว่าคอนเทนเนอร์ AI ทั้งหมดได้ยกเลิกสิทธิ์ของ Linux ทั้งหมด ยกเว้น CAP_SETUID, CAP_SETGID และสิทธิ์ที่ระบุไว้อย่างชัดเจนในเอกสารฐานความมั่นคงปลอดภัย                                               |   1   | D/V  |
| 4.1.2 | ตรวจสอบให้แน่ใจว่าโปรไฟล์ seccomp บล็อกการเรียก syscalls ทั้งหมด ยกเว้นที่อยู่ในรายการอนุญาตล่วงหน้าที่ได้รับการอนุมัติ โดยเมื่อเกิดการละเมิดจะทำให้คอนเทนเนอร์หยุดทำงานและสร้างการแจ้งเตือนด้านความปลอดภัยขึ้น |   1   | D/V  |
| 4.1.3 | ตรวจสอบให้แน่ใจว่างาน AI ทำงานบนระบบไฟล์รูทแบบอ่านอย่างเดียว, ใช้ tmpfs สำหรับข้อมูลชั่วคราว, และใช้โวลุ่มที่ตั้งชื่อสำหรับข้อมูลที่ต้องเก็บถาวรโดยมีตัวเลือกการเมานต์ noexec ถูกบังคับใช้                      |   2   | D/V  |
| 4.1.4 | ตรวจสอบให้แน่ใจว่าการตรวจสอบแบบรันไทม์ที่ใช้ eBPF (เช่น Falco, Tetragon หรือเทียบเท่า) สามารถตรวจจับความพยายามในการยกระดับสิทธิ์และฆ่ากระบวนการที่ละเมิดโดยอัตโนมัติภายในเวลาตอบสนองที่องค์กรกำหนดไว้           |   2   | D/V  |
| 4.1.5 | ตรวจสอบให้แน่ใจว่างานโหลด AI ที่มีความเสี่ยงสูงดำเนินการในสภาพแวดล้อมที่แยกฮาร์ดแวร์ (เช่น Intel TXT, AMD SVM หรือโหนด bare-metal เฉพาะ) พร้อมด้วยการตรวจสอบการรับรองความถูกต้อง (attestation verification)     |   3   | D/V  |

| :---: | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | :---: | :--: |

| 4.2.3 | ตรวจสอบให้แน่ใจว่าอิมเมจคอนเทนเนอร์ฝัง CycloneDX หรือ SPDX SBOMs และถูกลงนามด้วย Cosign ก่อนการส่งขึ้นรีจิสทรีย์ โดยอิมเมจที่ไม่ได้ลงนามจะถูกปฏิเสธในขั้นตอนการติดตั้งใช้งาน               |   2   | D/V  |
| 4.2.4 | ตรวจสอบให้แน่ใจว่า pipeline ของ CI/CD ใช้โทเค็น OIDC จาก HashiCorp Vault, บทบาท IAM ของ AWS หรือ Managed Identity ของ Azure โดยมีอายุการใช้งานไม่เกินขีดจำกัดตามนโยบายความปลอดภัยขององค์กร |   2   | D/V  |
| 4.2.5 | ตรวจสอบให้แน่ใจว่าลายเซ็น Cosign และการพิสูจน์แหล่งที่มาของ SLSA ถูกตรวจสอบในระหว่างกระบวนการปรับใช้ก่อนการทำงานของคอนเทนเนอร์ และข้อผิดพลาดในการตรวจสอบจะทำให้การปรับใช้ล้มเหลว           |   2   | D/V  |
| 4.2.6 | ตรวจสอบให้แน่ใจว่าสภาพแวดล้อมการสร้างทำงานในคอนเทนเนอร์หรือเครื่องเสมือนชั่วคราวที่ไม่มีการจัดเก็บข้อมูลถาวรและมีการแยกเครือข่ายจาก VPC ของระบบการผลิต                                     |   2   | D/V  |

## C4.3 ความปลอดภัยของเครือข่ายและการควบคุมการเข้าถึง

ดำเนินการเครือข่ายแบบศูนย์ความเชื่อถือพร้อมนโยบายปฏิเสธเริ่มต้นและการสื่อสารที่เข้ารหัส

|   #   | Description                                                                                                                                                                                       | Level | Role |
| :---: | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 4.3.1 | ตรวจสอบให้แน่ใจว่า Kubernetes NetworkPolicies หรือระบบเทียบเท่าใด ๆ ได้ดำเนินการนโยบายการปฏิเสธค่าเริ่มต้นในการรับเข้า/ออกด้วยกฎอนุญาตที่ระบุอย่างชัดเจนสำหรับพอร์ตที่จำเป็น (443, 8080, เป็นต้น) |   1   | D/V  |
| 4.3.2 | ตรวจสอบให้แน่ใจว่า SSH (พอร์ต 22), RDP (พอร์ต 3389), และจุดสิ้นสุดของข้อมูลเมตาในคลาวด์ (169.254.169.254) ถูกบล็อกหรือจำเป็นต้องมีการยืนยันตัวตนด้วยใบรับรอง                                      |   1   | D/V  |
| 4.3.3 | ตรวจสอบให้แน่ใจว่าการรับส่งข้อมูลออกถูกกรองผ่านพร็อกซี HTTP/HTTPS (เช่น Squid, Istio หรือเกตเวย์ NAT ของคลาวด์) โดยมีรายการอนุญาตโดเมนและบันทึกคำขอบล็อกไว้                                       |   2   | D/V  |
| 4.3.4 | ตรวจสอบให้แน่ใจว่าการสื่อสารระหว่างบริการใช้ mutual TLS โดยมีการเปลี่ยนใบรับรองตามนโยบายขององค์กรและมีการบังคับใช้การตรวจสอบความถูกต้องของใบรับรอง (ไม่มีการใช้แฟล็ก skip-verify)                 |   2   | D/V  |
| 4.3.5 | ตรวจสอบให้แน่ใจว่าโครงสร้างพื้นฐาน AI ทำงานใน VPCs/VNets ที่จัดสรรเฉพาะโดยไม่มีการเข้าถึงอินเทอร์เน็ตโดยตรง และสื่อสารผ่านเกตเวย์ NAT หรือโฮสต์บาสชันเท่านั้น                                     |   2   | D/V  |

## C4.4 การจัดการความลับและกุญแจเข้ารหัส

ปกป้องข้อมูลรับรองผ่านการจัดเก็บที่สนับสนุนโดยฮาร์ดแวร์และการหมุนเวียนอัตโนมัติด้วยการเข้าถึงแบบศูนย์ความเชื่อถือ (zero-trust)

|   #   | Description                                                                                                                                                                               | Level | Role |
| :---: | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 4.4.1 | ตรวจสอบให้แน่ใจว่าข้อมูลลับถูกเก็บไว้ใน HashiCorp Vault, AWS Secrets Manager, Azure Key Vault หรือ Google Secret Manager โดยมีการเข้ารหัสข้อมูลที่จัดเก็บด้วย AES-256                     |   1   | D/V  |
| 4.4.2 | ตรวจสอบให้แน่ใจว่ากุญแจเข้ารหัสถูกสร้างขึ้นใน HSM ระดับ FIPS 140-2 Level 2 (AWS CloudHSM, Azure Dedicated HSM) โดยมีการหมุนเวียนกุญแจตามนโยบายการเข้ารหัสขององค์กร                        |   1   | D/V  |
| 4.4.3 | ตรวจสอบให้แน่ใจว่าการหมุนเวียนความลับเป็นแบบอัตโนมัติพร้อมการปรับใช้ที่ไม่มีเวลาหยุดทำงาน และมีการหมุนเวียนทันทีเมื่อเกิดการเปลี่ยนแปลงบุคลากรหรือเหตุการณ์ด้านความปลอดภัย                |   2   | D/V  |
| 4.4.4 | ตรวจสอบให้แน่ใจว่าภาพคอนเทนเนอร์ถูกสแกนด้วยเครื่องมือ (GitLeaks, TruffleHog หรือ detect-secrets) เพื่อบล็อกการสร้างที่มีคีย์ API รหัสผ่าน หรือใบรับรอง                                    |   2   | D/V  |
| 4.4.5 | ตรวจสอบให้แน่ใจว่าการเข้าถึงความลับในสภาพแวดล้อมการผลิตต้องใช้ MFA กับฮาร์ดแวร์โทเค็น (YubiKey, FIDO2) และบันทึกโดยบันทึกตรวจสอบที่ไม่สามารถแก้ไขได้พร้อมทั้งระบุตัวตนผู้ใช้และเวลาประทับ |   2   | D/V  |
| 4.4.6 | ตรวจสอบให้แน่ใจว่าสารลับถูกฉีดผ่าน Kubernetes secrets, โวลุ่มที่ติดตั้ง, หรือ init containers และตรวจสอบให้แน่ใจว่าสารลับไม่เคยถูกฝังอยู่ในตัวแปรสภาพแวดล้อมหรืออิมเมจ                    |   2   | D/V  |

## C4.5 การจำลองสภาพแวดล้อมงาน AI และการตรวจสอบความถูกต้อง

แยกโมเดล AI ที่ไม่น่าเชื่อถือให้อยู่ในสภาพแวดล้อมแซนด์บ็อกซ์ที่ปลอดภัยพร้อมการวิเคราะห์พฤติกรรมอย่างครอบคลุม

|   #   | Description                                                                                                                                                                                                                                     | Level | Role |
| :---: | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 4.5.1 | ตรวจสอบให้แน่ใจว่าโมเดล AI ภายนอกทำงานใน gVisor, microVMs (เช่น Firecracker, CrossVM) หรือคอนเทนเนอร์ Docker โดยใช้ตัวเลือก --security-opt=no-new-privileges และแฟล็ก --read-only                                                               |   1   | D/V  |
| 4.5.2 | ตรวจสอบว่าสภาพแวดล้อมแซนด์บ็อกซ์ไม่มีการเชื่อมต่อเครือข่าย (--network=none) หรืออนุญาตให้เข้าถึงเฉพาะ localhost เท่านั้นโดยมีการบล็อกคำขอภายนอกทั้งหมดด้วยกฎ iptables                                                                           |   1   | D/V  |
| 4.5.3 | ตรวจสอบให้แน่ใจว่าการตรวจสอบความถูกต้องของโมเดล AI รวมถึงการทดสอบทีมแดงอัตโนมัติที่มีขอบเขตการทดสอบและการวิเคราะห์พฤติกรรมที่กำหนดโดยองค์กรเพื่อการตรวจจับประตูหลัง                                                                             |   2   | D/V  |
| 4.5.4 | ตรวจสอบให้แน่ใจว่าก่อนที่โมเดล AI จะได้รับการเลื่อนขั้นไปยังขั้นตอนการใช้งานจริง ผลลัพธ์จาก sandbox ของโมเดลนั้นได้รับการลงนามทางคริปโตกราฟีโดยเจ้าหน้าที่รักษาความปลอดภัยที่ได้รับอนุญาตและถูกจัดเก็บไว้ในบันทึกการตรวจสอบที่ไม่สามารถแก้ไขได้ |   2   | D/V  |
| 4.5.5 | ตรวจสอบให้แน่ใจว่าสภาพแวดล้อม sandbox ถูกทำลายและสร้างขึ้นใหม่จากภาพต้นแบบ (golden images) ระหว่างการประเมินผล พร้อมกับการล้างข้อมูลระบบไฟล์และหน่วยความจำอย่างสมบูรณ์                                                                          |   2   | D/V  |

## C4.6 การตรวจสอบความปลอดภัยโครงสร้างพื้นฐาน

| 4.6.1 | ตรวจสอบให้แน่ใจว่าภาพคอนเทนเนอร์ได้รับการสแกนตามตารางเวลาขององค์กร โดยมีช่องโหว่ที่ร้ายแรง (CRITICAL) เป็นตัวบล็อกการปรับใช้ตามเกณฑ์ความเสี่ยงขององค์กร                                                             |   1   | D/V  |

## C4.20 โครงสร้างพื้นฐานการคำนวณที่รักษาความเป็นส่วนตัว

ดำเนินการควบคุมโครงสร้างพื้นฐานเพื่อการคำนวณที่รักษาความเป็นส่วนตัวเพื่อปกป้องข้อมูลที่ละเอียดอ่อนในระหว่างการประมวลผลและวิเคราะห์ AI

|   #    | Description                                                                                                                                                                                                                                       | Level | Role |
| :----: | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 4.20.1 | ตรวจสอบให้แน่ใจว่าโครงสร้างพื้นฐานการเข้ารหัสฮอโมมอร์ฟิกช่วยให้สามารถคำนวณแบบเข้ารหัสบนงาน AI ที่มีความละเอียดอ่อนได้ พร้อมกับการตรวจสอบความถูกต้องของการเข้ารหัสทางคริปโตกราฟีและการตรวจจับประสิทธิภาพการทำงาน                                   |   3   | D/V  |
| 4.20.2 | ตรวจสอบให้แน่ใจว่าระบบการดึงข้อมูลส่วนตัว (private information retrieval) สามารถทำให้การค้นหาฐานข้อมูลทำได้โดยไม่เปิดเผยรูปแบบการค้นหา พร้อมด้วยการป้องกันเชิงเข้ารหัสของรูปแบบการเข้าถึงข้อมูล                                                   |   3   | D/V  |
| 4.20.3 | ตรวจสอบให้แน่ใจว่าโพรโทคอลการคำนวณแบบมัลติพาร์ตี้ที่ปลอดภัยช่วยให้การอนุมาน AI ที่รักษาความเป็นส่วนตัวโดยไม่เปิดเผยข้อมูลอินพุตแต่ละรายการหรือการคำนวณระหว่างขั้นตอน                                                                              |   3   | D/V  |

| 4.20.5 | ตรวจสอบให้แน่ใจว่าประสิทธิภาพการประมวลผลที่รักษาความเป็นส่วนตัวได้รับการปรับให้เหมาะสมผ่านการจัดกลุ่มข้อมูล (batching), การเก็บข้อมูลชั่วคราว (caching) และการเร่งความเร็วด้วยฮาร์ดแวร์ พร้อมกับรักษาการรับประกันความปลอดภัยทางคริปโตกราฟีไว้เสมอ |   3   | D/V  |

## C4.15 การผสานระบบความปลอดภัยของเฟรมเวิร์กเอเจนต์คลาวด์และการติดตั้งแบบไฮบริด

| 4.15.2 | ตรวจสอบให้แน่ใจว่าขอบเขตความปลอดภัยของการปรับใช้งานแบบผสมผสานถูกกำหนดไว้อย่างชัดเจนพร้อมกับช่องทางการสื่อสารที่ถูกเข้ารหัส        |   2   | D/V  |
| 4.15.3 | ตรวจสอบให้แน่ใจว่าการเข้าถึงทรัพยากรคลาวด์รวมถึงการตรวจสอบแบบ zero-trust พร้อมกับการยืนยันตัวตนอย่างต่อเนื่อง                     |   2   | D/V  |
| 4.15.4 | ตรวจสอบให้แน่ใจว่าข้อกำหนดการจัดเก็บข้อมูลในภูมิภาคถูกบังคับใช้โดยการรับรองตำแหน่งที่จัดเก็บข้อมูลด้วยวิธีเข้ารหัสทางคณิตศาสตร์   |   3   | D/V  |
