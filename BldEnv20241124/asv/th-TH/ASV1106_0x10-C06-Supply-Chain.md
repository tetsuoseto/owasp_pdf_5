# C6 ความปลอดภัยของห่วงโซ่อุปทานสำหรับโมเดล กรอบงาน และข้อมูล

## วัตถุประสงค์ของการควบคุม

การโจมตีห่วงโซ่อุปทานของ AI ใช้ประโยชน์จากโมเดล เฟรมเวิร์ก หรือชุดข้อมูลของบุคคลที่สาม เพื่อฝังประตูหลัง อคติ หรือโค้ดที่สามารถถูกโจมตีได้ การควบคุมเหล่านี้ให้การติดตามแหล่งที่มาแบบครบวงจร การจัดการช่องโหว่ และการตรวจสอบ เพื่อปกป้องวงจรชีวิตของโมเดลทั้งหมด

---

## C6.1 การตรวจสอบและการระบุแหล่งที่มาของโมเดลที่ผ่านการฝึกอบรมล่วงหน้า

ประเมินและตรวจสอบแหล่งที่มา ใบอนุญาต และพฤติกรรมที่ซ่อนเร้นของโมเดลจากบุคคลที่สามก่อนการปรับแต่งหรือการใช้งานจริงใดๆ

|   #   | Description                                                                                                                                                 | Level | Role |
| :---: | ----------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 6.1.1 | ตรวจสอบให้แน่ใจว่าวัสดุของโมเดลจากบุคคลที่สามทุกชิ้นมีบันทึกแหล่งที่มาที่ลงนามระบุที่เก็บข้อมูลต้นทางและรหัสคอมมิต                                          |   1   | D/V  |
| 6.1.2 | ตรวจสอบว่าโมเดลถูกสแกนเพื่อตรวจหาชั้นที่เป็นอันตรายหรือทริกเกอร์ Trojan โดยใช้เครื่องมืออัตโนมัติก่อนนำเข้า                                                 |   1   | D/V  |
| 6.1.3 | ยืนยันว่าการปรับแต่งแบบทรานส์เฟอร์เลิร์นนิงสามารถผ่านการประเมินแบบย้อนศรเพื่อจับพฤติกรรมที่ซ่อนเร้นได้                                                      |   2   |  D   |
| 6.1.4 | ตรวจสอบให้แน่ใจว่าใบอนุญาตของโมเดล ป้ายกำกับการควบคุมการส่งออก และคำชี้แจงแหล่งกำเนิดข้อมูล ถูกบันทึกในรายการ ML-BOM                                        |   2   |  V   |
| 6.1.5 | ตรวจสอบให้แน่ใจว่าโมเดลที่มีความเสี่ยงสูง (น้ำหนักที่อัปโหลดสู่สาธารณะ ผู้สร้างที่ไม่ได้รับการยืนยัน) ยังคงถูกกักกันจนกว่าจะมีการตรวจสอบและอนุมัติโดยมนุษย์ |   3   | D/V  |

## C6.2 การสแกนกรอบงานและไลบรารี

สแกนเฟรมเวิร์กและไลบรารี ML อย่างต่อเนื่องเพื่อหาช่องโหว่ด้านความปลอดภัย (CVEs) และโค้ดที่เป็นอันตรายเพื่อรักษาความปลอดภัยของสแตกเวลารันไทม์

|   #   | Description                                                                                                                              | Level | Role |
| :---: | ---------------------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 6.2.1 | ตรวจสอบให้แน่ใจว่า CI pipeline ทำการสแกนหาความขึ้นต่อกันของไลบรารีในเฟรมเวิร์ก AI และไลบรารีสำคัญๆ                                       |   1   | D/V  |
| 6.2.2 | ตรวจสอบให้แน่ใจว่าช่องโหว่ร้ายแรง (CVSS ≥ 7.0) จะถูกบล็อกไม่ให้ส่งเสริมไปยังภาพผลิตภัณฑ์                                                 |   1   | D/V  |
| 6.2.3 | ตรวจสอบให้แน่ใจว่าการวิเคราะห์โค้ดแบบสแตติกทำงานบนไลบรารี ML ที่ถูกแยกออกมา (forked) หรือที่นำเข้ามาเป็นส่วนหนึ่งของโค้ด (vendored) แล้ว |   2   |  D   |
| 6.2.4 | ตรวจสอบให้แน่ใจว่าข้อเสนอการอัปเกรดเฟรมเวิร์กมีการประเมินผลกระทบด้านความปลอดภัยโดยอ้างอิงจากแหล่งข้อมูล CVE สาธารณะ                      |   2   |  V   |
| 6.2.5 | ยืนยันว่าตัวตรวจจับเวลารันไทม์แจ้งเตือนเมื่อมีการโหลดไลบรารีไดนามิกที่ไม่คาดคิดซึ่งเบี่ยงเบนจาก SBOM ที่ลงนามไว้                         |   3   |  V   |

## C6.3 การตรึงและการตรวจสอบการพึ่งพา

ตรึงการพึ่งพาทุกอย่างไว้กับไดเจสต์ที่ไม่เปลี่ยนแปลง และทำการสร้างซ้ำเพื่อรับประกันว่าอาร์ติแฟกต์จะเหมือนกันและปราศจากการปลอมแปลง

|   #   | Description                                                                                                                    | Level | Role |
| :---: | ------------------------------------------------------------------------------------------------------------------------------ | :---: | :--: |
| 6.3.1 | ตรวจสอบให้แน่ใจว่าตัวจัดการแพ็กเกจทั้งหมดบังคับใช้การระบุเวอร์ชันอย่างเข้มงวดผ่านไฟล์ล็อก (lockfiles)                          |   1   | D/V  |
| 6.3.2 | ตรวจสอบให้แน่ใจว่าใช้ดิจิทที่ไม่เปลี่ยนแปลงแทนแท็กที่เปลี่ยนแปลงได้ในข้อมูลอ้างอิงคอนเทนเนอร์                                  |   1   | D/V  |
| 6.3.3 | ตรวจสอบให้แน่ใจว่าการตรวจสอบการสร้างซ้ำ (reproducible-build checks) เปรียบเทียบแฮชข้ามการรัน CI เพื่อยืนยันผลลัพธ์ที่เหมือนกัน |   2   |  D   |
| 6.3.4 | ตรวจสอบให้แน่ใจว่าการรับรองการสร้างถูกเก็บไว้เป็นเวลา 18 เดือนเพื่อการตรวจสอบย้อนกลับได้                                       |   2   |  V   |
| 6.3.5 | ตรวจสอบให้แน่ใจว่าการพึ่งพาที่หมดอายุจะกระตุ้นให้เกิดการส่งคำขอปรับปรุงอัตโนมัติเพื่ออัปเดตหรือแยกเวอร์ชันที่ถูกตรึงไว้        |   3   |  D   |

## C6.4 การบังคับใช้แหล่งที่มาที่เชื่อถือได้

อนุญาตให้ดาวน์โหลดอาร์ติแฟกต์เฉพาะจากแหล่งที่ได้รับการตรวจสอบทางคริปโตกราฟีและได้รับการอนุมัติจากองค์กรเท่านั้น และบล็อกทุกอย่างอื่น ๆ

|   #   | Description                                                                                                                                                               | Level | Role |
| :---: | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 6.4.1 | ตรวจสอบให้แน่ใจว่าน้ำหนักของแบบจำลอง ชุดข้อมูล และคอนเทนเนอร์ ถูกดาวน์โหลดเฉพาะจากโดเมนที่ได้รับอนุญาตหรือทะเบียนภายในเท่านั้น                                            |   1   | D/V  |
| 6.4.2 | ตรวจสอบให้แน่ใจว่าลายเซ็น Sigstore/Cosign ยืนยันตัวตนผู้เผยแพร่ก่อนที่อาร์ติแฟกต์จะถูกแคชในเครื่องlocal                                                                   |   1   | D/V  |
| 6.4.3 | ตรวจสอบให้แน่ใจว่า proxy ทางออก (egress proxies) บล็อกการดาวน์โหลดอาร์ติแฟกต์ที่ไม่ได้รับการยืนยันตัวตน เพื่อบังคับใช้นโยบายแหล่งที่มาเชื่อถือได้ (trusted-source policy) |   2   |  D   |
| 6.4.4 | ตรวจสอบให้แน่ใจว่ารายการอนุญาตในที่เก็บข้อมูลได้รับการทบทวนทุกไตรมาส พร้อมหลักฐานการอ้างเหตุผลทางธุรกิจสำหรับแต่ละรายการ                                                  |   2   |  V   |
| 6.4.5 | ยืนยันว่าการละเมิดนโยบายจะทำให้เกิดการกักกันอาร์ติแฟกต์และย้อนกลับการทำงานของพึ่งพิงในพายไลน์                                                                             |   3   |  V   |

## C6.5 การประเมินความเสี่ยงของชุดข้อมูลจากบุคคลที่สาม

ประเมินชุดข้อมูลภายนอกเพื่อหาการปนเปื้อน อคติ และความสอดคล้องกับกฎหมาย และติดตามตรวจสอบตลอดวงจรชีวิตของชุดข้อมูลเหล่านั้น

|   #   | Description                                                                                                                  | Level | Role |
| :---: | ---------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 6.5.1 | ตรวจสอบให้แน่ใจว่าชุดข้อมูลภายนอกได้รับการประเมินความเสี่ยงการปนเปื้อน (เช่น การสร้างลายนิ้วมือข้อมูล, การตรวจจับค่าผิดปกติ) |   1   | D/V  |
| 6.5.2 | ตรวจสอบให้แน่ใจว่าเมตริกความลำเอียง (ความเท่าเทียมทางประชากรศาสตร์, โอกาสที่เท่าเทียมกัน) ถูกคำนวณก่อนการอนุมัติชุดข้อมูล    |   1   |  D   |
| 6.5.3 | ตรวจสอบให้แน่ใจว่าข้อมูลต้นกำเนิดและข้อกำหนดใบอนุญาตสำหรับชุดข้อมูลถูกบันทึกในรายการ ML-BOM                                  |   2   |  V   |
| 6.5.4 | ตรวจสอบให้แน่ใจว่าการตรวจสอบเป็นระยะสามารถตรวจจับการเปลี่ยนแปลงหรือความผิดปกติในชุดข้อมูลที่โฮสต์อยู่ได้                     |   2   |  V   |
| 6.5.5 | ตรวจสอบให้แน่ใจว่าสารสนเทศที่ไม่อนุญาต (ลิขสิทธิ์, ข้อมูลส่วนบุคคล) ถูกลบออกผ่านการลบอัตโนมัติก่อนการฝึกอบรมแล้ว             |   3   |  D   |

## C6.6 การตรวจสอบการโจมตีห่วงโซ่อุปทาน

ตรวจจับภัยคุกคามในห่วงโซ่อุปทานตั้งแต่เนิ่นๆ ผ่านฟีด CVE การวิเคราะห์บันทึกการตรวจสอบ และการจำลองทีมแดง

|   #   | Description                                                                                                                                   | Level | Role |
| :---: | --------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 6.6.1 | ตรวจสอบให้แน่ใจว่าบันทึกการตรวจสอบ CI/CD ถูกส่งสตรีมไปยัง SIEM เพื่อการตรวจจับการดึงแพ็กเกจที่ผิดปกติหรือขั้นตอนการสร้างที่ถูกดัดแปลง         |   1   |  V   |
| 6.6.2 | ตรวจสอบให้แน่ใจว่าแผนการตอบสนองเหตุการณ์มีขั้นตอนการย้อนกลับสำหรับโมเดลหรือไลบรารีที่ถูกโจมตี                                                 |   2   |  D   |
| 6.6.3 | ตรวจสอบให้แน่ใจว่าการเสริมข้อมูลข่าวกรองภัยคุกคามกำกับแท็กดัชนีชี้วัดเฉพาะ ML (เช่น IoC การโจมตีแบบปนเปื้อนโมเดล) ในกระบวนการคัดกรองแจ้งเตือน |   3   |  V   |

## C6.7 รายการวัสดุสำหรับแบบจำลองแมชชีนเลิร์นนิง (ML-BOM) สำหรับวัตถุของแบบจำลอง

สร้างและลงนาม SBOMs เฉพาะทาง ML รายละเอียด (ML-BOMs) เพื่อให้ผู้ใช้ปลายทางสามารถตรวจสอบความสมบูรณ์ของส่วนประกอบในเวลาที่ติดตั้งใช้งานได้

|   #   | Description                                                                                                                           | Level | Role |
| :---: | ------------------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 6.7.1 | ตรวจสอบให้แน่ใจว่าอาร์ติแฟกต์ของโมเดลแต่ละตัวเผยแพร่ ML-BOM ที่ระบุชุดข้อมูล น้ำหนัก พารามิเตอร์ที่ตั้งไว้ล่วงหน้า และใบอนุญาตทั้งหมด |   1   | D/V  |
| 6.7.2 | ตรวจสอบให้แน่ใจว่าการสร้าง ML-BOM และการลงนาม Cosign เป็นไปโดยอัตโนมัติใน CI และเป็นข้อกำหนดสำหรับการรวมโค้ด                          |   1   | D/V  |
| 6.7.3 | ตรวจสอบให้แน่ใจว่าการตรวจสอบความสมบูรณ์ของ ML-BOM จะทำให้การสร้างล้มเหลวหากมีข้อมูลเมตาใด ๆ ของส่วนประกอบ (แฮช, สิทธิ์ใช้งาน) หายไป   |   2   |  D   |
| 6.7.4 | ตรวจสอบให้แน่ใจว่าผู้ใช้ปลายทางสามารถสอบถาม ML-BOMs ผ่าน API เพื่อยืนยันความถูกต้องของโมเดลที่นำเข้าในขณะติดตั้งใช้งานได้             |   2   |  V   |
| 6.7.5 | ตรวจสอบให้แน่ใจว่า ML-BOMs มีการควบคุมเวอร์ชันและเปรียบเทียบความแตกต่างเพื่อค้นหาการแก้ไขที่ไม่ได้รับอนุญาต                           |   3   |  V   |
