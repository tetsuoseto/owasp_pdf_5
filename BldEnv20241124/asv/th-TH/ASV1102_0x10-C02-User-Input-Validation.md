# การตรวจสอบความถูกต้องของข้อมูลที่ผู้ใช้ป้อนในระดับ C2

## วัตถุประสงค์ในการควบคุม

การตรวจสอบความถูกต้องของข้อมูลที่ผู้ใช้ป้อนอย่างเข้มงวดเป็นแนวป้องกันแรกต่อการโจมตีที่สร้างความเสียหายร้ายแรงต่อระบบปัญญาประดิษฐ์ การโจมตีแบบ prompt injection สามารถลบล้างคำสั่งของระบบ รั่วไหลข้อมูลที่ละเอียดอ่อน หรือชักนำโมเดลไปสู่พฤติกรรมที่ไม่ได้รับอนุญาต หากไม่มีการกรองเฉพาะและลำดับชั้นของคำสั่งงาน งานวิจัยแสดงให้เห็นว่า jailbreak แบบ "multi-shot" ที่ใช้หน้าต่างบริบทยาวมาก จะได้ผล การโจมตีแบบปรับแต่งซ่อนเร้นอย่างละเอียด เช่น การแทนที่ด้วย homoglyph หรือ leetspeak สามารถเปลี่ยนแปลงการตัดสินใจของโมเดลได้อย่างเงียบเชียบด้วยเช่นกัน

---

## C2.1 การป้องกันการฉีดคำสั่ง

การโจมตีด้วยการแทรกคำสั่ง (Prompt injection) เป็นหนึ่งในความเสี่ยงสูงสุดสำหรับระบบปัญญาประดิษฐ์ (AI) การป้องกันวิธีนี้ใช้การผสมผสานระหว่างตัวกรองรูปแบบคงที่ (static pattern filters), ตัวจัดหมวดหมู่แบบไดนามิก (dynamic classifiers) และการบังคับใช้ลำดับขั้นของคำสั่ง (instruction hierarchy enforcement)

|   #   | Description                                                                                                                                                                                                              | Level | Role |
| :---: | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | :---: | :--: |
| 2.1.1 | ตรวจสอบให้แน่ใจว่าการป้อนข้อมูลของผู้ใช้ถูกกรองกับห้องสมุดที่อัปเดตอย่างต่อเนื่องของรูปแบบการแทรกคำสั่งที่รู้จัก (คีย์เวิร์ดเจลเบรค, "ละเว้นคำสั่งก่อนหน้า", โซ่บทบาทสมมติ, การโจมตี HTML/URL แบบอ้อม)                   |   1   | D/V  |
| 2.1.2 | ตรวจสอบให้แน่ใจว่าระบบบังคับใช้ลำดับชั้นของคำแนะนำโดยที่ข้อความจากระบบหรือผู้พัฒนาจะมีอำนาจเหนือคำสั่งของผู้ใช้ แม้หลังจากขยายหน้าต่างบริบทแล้วก็ตาม                                                                     |   1   | D/V  |
| 2.1.3 | ตรวจสอบให้แน่ใจว่าการทดสอบประเมินความเป็นศัตรู (เช่น prompt แบบ "many-shot" ของทีมแดง) ถูกดำเนินการก่อนการปล่อยโมเดลหรือแม่แบบ prompt ทุกครั้ง โดยมีเกณฑ์อัตราความสำเร็จและตัวบล็อกอัตโนมัติสำหรับการถดถอยของประสิทธิภาพ |   2   | D/V  |
| 2.1.4 | ตรวจสอบให้แน่ใจว่าคำสั่งที่มาจากเนื้อหาของบุคคลที่สาม (เว็บเพจ, ไฟล์ PDF, อีเมล) ถูกทำความสะอาดในบริบทการแยกวิเคราะห์แบบแยกส่วนก่อนที่จะถูกรวมเข้าเป็นคำสั่งหลัก                                                         |   2   |  D   |
| 2.1.5 | ตรวจสอบให้แน่ใจว่าการอัปเดตกฎการกรองคำสั่ง, เวอร์ชันของโมเดลตัวจัดประเภท และการเปลี่ยนแปลงรายชื่อบล็อกทั้งหมดมีการควบคุมเวอร์ชันและสามารถตรวจสอบได้                                                                      |   3   | D/V  |

## C2.2 ความต้านทานต่อการโจมตีด้วยตัวอย่างที่เป็นการท้าทาย

แบบจำลองการประมวลผลภาษาธรรมชาติ (NLP) ยังคงมีความเปราะบางต่อการเปลี่ยนแปลงเล็กน้อยในระดับตัวอักษรหรือคำที่มนุษย์มักไม่สังเกตเห็นแต่แบบจำลองมักจะจำแนกผิดพลาด

|   #   | Description                                                                                                                                                                                                           | Level | Role |
| :---: | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 2.2.1 | ตรวจสอบให้แน่ใจว่าขั้นตอนการปรับรูปแบบข้อมูลเบื้องต้น (Unicode NFC, การจับคู่โฮโมไกลฟ์, การตัดช่องว่าง) ทำงานก่อนการแบ่งโทเค็น                                                                                        |   1   |  D   |
| 2.2.2 | ตรวจสอบว่า การตรวจจับความผิดปกติทางสถิติ ระบุอินพุตที่มีระยะห่างการแก้ไขสูงผิดปกติต่อเกณฑ์ของภาษา, การทำซ้ำของโทเค็นมากเกินไป หรือระยะห่างของการฝังตัวที่ผิดปกติหรือไม่                                               |   2   | D/V  |
| 2.2.3 | ตรวจสอบให้แน่ใจว่าเส้นทางการอนุมานรองรับรุ่นโมเดลที่แข็งแกร่งด้วยการฝึกอบรมแบบต่อต้าน (adversarial-training) แบบเลือกใช้ได้ หรือชั้นป้องกัน (เช่น การสุ่ม การกลั่นกรองเชิงป้องกัน) สำหรับจุดสิ้นสุดที่มีความเสี่ยงสูง |   2   |  D   |
| 2.2.4 | ตรวจสอบให้แน่ใจว่าข้อมูลป้อนเข้าที่สงสัยว่าเป็นการโจมตีถูกแยกกักไว้, จดบันทึกพร้อมข้อมูลเพย์โหลดเต็มรูปแบบ (หลังจากการลบข้อมูลส่วนบุคคลที่ระบุตัวได้)                                                                 |   2   |  V   |
| 2.2.5 | ตรวจสอบให้แน่ใจว่ามาตรวัดความทนทาน (อัตราความสำเร็จของชุดโจมตีที่รู้จัก) ถูกติดตามตามเวลาและการถดถอยจะกระตุ้นการหยุดปล่อยซอฟต์แวร์                                                                                    |   3   | D/V  |

## C2.3 การตรวจสอบความถูกต้องของสคีมา ชนิดข้อมูล และความยาว

การโจมตี AI ที่ใช้ข้อมูลนำเข้าที่ผิดรูปแบบหรือมีขนาดใหญ่เกินไป อาจทำให้เกิดข้อผิดพลาดในการแยกวิเคราะห์ข้อมูล (parsing errors), การรั่วไหลของพรอมพ์ข้ามฟิลด์ (prompt spillage across fields) และการใช้ทรัพยากรเกินจำเป็น (resource exhaustion) การบังคับใช้สคีมาอย่างเข้มงวดเป็นสิ่งจำเป็นเมื่อทำการเรียกใช้เครื่องมือแบบกำหนดได้ (deterministic tool calls)

|   #   | Description                                                                                                                                                                                                                | Level | Role |
| :---: | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 2.3.1 | ตรวจสอบให้แน่ใจว่าทุกจุดสิ้นสุดของการเรียก API หรือฟังก์ชันกำหนดสคีมารูปแบบข้อมูลนำเข้าอย่างชัดเจน (JSON Schema, Protobuf หรือรูปแบบมัลติโมดอลที่เทียบเท่า) และตรวจสอบความถูกต้องของข้อมูลนำเข้าก่อนการประกอบคำสั่ง prompt |   1   |  D   |
| 2.3.2 | ตรวจสอบให้แน่ใจว่าข้อมูลนำเข้าที่เกินขีดจำกัดสูงสุดของโทเค็นหรือไบต์จะถูกปฏิเสธพร้อมกับข้อผิดพลาดที่ปลอดภัยและไม่ถูกตัดทิ้งอย่างเงียบๆ                                                                                     |   1   | D/V  |
| 2.3.3 | ตรวจสอบให้แน่ใจว่าการตรวจสอบประเภท (เช่น ช่วงตัวเลข ค่าของ enum ประเภท MIME สำหรับภาพ/เสียง) ถูกบังคับใช้ที่ฝั่งเซิร์ฟเวอร์ ไม่ใช่แค่ในโค้ดฝั่งไคลเอนต์เท่านั้น                                                            |   2   | D/V  |
| 2.3.4 | ตรวจสอบให้แน่ใจว่า semantic validators (เช่น JSON Schema) ทำงานภายในเวลาคงที่เพื่อป้องกันการโจมตีแบบ DoS จากอัลกอริทึม                                                                                                     |   2   |  D   |
| 2.3.5 | ตรวจสอบให้แน่ใจว่าการตรวจสอบความถูกต้องที่ล้มเหลวถูกบันทึกด้วยส่วนของข้อมูลที่ถูกลบออกและรหัสข้อผิดพลาดที่ชัดเจนเพื่อช่วยในการวิเคราะห์ด้านความปลอดภัย                                                                     |   3   |  V   |

## C2.4 การคัดกรองเนื้อหาและนโยบาย

นักพัฒนาควรสามารถตรวจจับคำสั่งที่ถูกต้องตามไวยากรณ์ซึ่งร้องขอเนื้อหาที่ไม่ได้รับอนุญาต (เช่น คำสั่งที่ผิดกฎหมาย, วาจาสร้างความเกลียดชัง, และข้อความที่มีลิขสิทธิ์) และป้องกันไม่ให้คำสั่งเหล่านั้นแพร่กระจายออกไปได้

|   #   | Description                                                                                                                                                                                                                            | Level | Role |
| :---: | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 2.4.1 | ตรวจสอบให้แน่ใจว่าเครื่องมือจำแนกเนื้อหา (แบบ zero shot หรือแบบ fine tuned) ประเมินคะแนนสำหรับแต่ละข้อมูลเข้าในหมวดความรุนแรง การทำร้ายตนเอง ความเกลียดชัง เนื้อหาทางเพศ และคำขอที่ผิดกฎหมาย โดยมีการตั้งค่าระดับคะแนนที่สามารถปรับได้ |   1   |  D   |
| 2.4.2 | ตรวจสอบให้แน่ใจว่าข้อมูลนำเข้าที่ละเมิดนโยบายจะได้รับการปฏิเสธที่เป็นมาตรฐานหรือการทำงานที่ปลอดภัยเพื่อป้องกันไม่ให้ส่งต่อไปยังการเรียกใช้ LLM ในขั้นตอนถัดไป                                                                          |   1   | D/V  |
| 2.4.3 | ตรวจสอบให้แน่ใจว่ารูปแบบการคัดกรองหรือชุดกฎมีการฝึกสอนใหม่/อัปเดตอย่างน้อยไตรมาสละครั้ง โดยรวมรูปแบบการเจลเบรคหรือการเลี่ยงนโยบายที่พบใหม่เข้ามาด้วย                                                                                   |   2   |  D   |
| 2.4.4 | ตรวจสอบให้แน่ใจว่าการคัดกรองเคารพนโยบายเฉพาะผู้ใช้ (อายุ ข้อจำกัดทางกฎหมายตามภูมิภาค) ผ่านกฎที่อิงตามแอตทริบิวต์ซึ่งถูกแก้ไขในเวลาที่มีการร้องขอ                                                                                       |   2   |  D   |
| 2.4.5 | ตรวจสอบให้แน่ใจว่าสมุดบันทึกการคัดกรองมีคะแนนความมั่นใจของตัวจัดหมวดหมู่และแท็กหมวดหมู่นโยบายสำหรับการเชื่อมโยง SOC และการเล่นซ้ำของทีมแดงในอนาคต                                                                                      |   3   |  V   |

## C2.5 การจำกัดอัตราการรับข้อมูลเข้าและการป้องกันการละเมิด

นักพัฒนาควรป้องกันการใช้ในทางที่ผิด การใช้ทรัพยากรเกินจำเป็น และการโจมตีอัตโนมัติต่อระบบปัญญาประดิษฐ์โดยการจำกัดอัตราการป้อนข้อมูลและตรวจจับรูปแบบการใช้งานที่ผิดปกติ

|   #   | Description                                                                                                                                                                                                      | Level | Role |
| :---: | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 2.5.1 | ตรวจสอบให้แน่ใจว่ามีการบังคับใช้อัตราจำกัดต่อผู้ใช้ ต่อ IP และต่อคีย์ API สำหรับทุกจุดเชื่อมต่ออินพุต                                                                                                            |   1   | D/V  |
| 2.5.2 | ตรวจสอบให้แน่ใจว่าการตั้งค่าขีดจำกัดอัตราการส่งข้อมูลแบบระเบิด (burst) และแบบต่อเนื่อง (sustained) ได้รับการปรับแต่งเพื่อป้องกันการโจมตีแบบปฏิเสธการให้บริการ (DoS) และการโจมตีแบบเดารหัสผ่าน (brute force) แล้ว |   2   | D/V  |
| 2.5.3 | ตรวจสอบให้แน่ใจว่ารูปแบบการใช้งานที่ผิดปกติ (เช่น คำขอที่ส่งอย่างรวดเร็ว, การป้อนข้อมูลอย่างล้นหลาม) ทำให้เกิดการบล็อกอัตโนมัติหรือการยกระดับการจัดการ                                                           |   2   | D/V  |
| 2.5.4 | ตรวจสอบให้แน่ใจว่าบันทึกการป้องกันการละเมิดถูกเก็บรักษาและทบทวนเพื่อหารูปแบบการโจมตีที่เกิดขึ้นใหม่                                                                                                              |   3   |  V   |

## C2.6 การตรวจสอบความถูกต้องของข้อมูลนำเข้าหลายรูปแบบ

ระบบปัญญาประดิษฐ์ควรรวมการตรวจสอบที่เข้มงวดสำหรับข้อมูลนำเข้าแบบไม่ใช่ข้อความ (ภาพ เสียง ไฟล์) เพื่อป้องกันการฉีดข้อมูล การหลบเลี่ยง หรือการใช้งานทรัพยากรเกินควร

|   #   | Description                                                                                                                                  | Level | Role |
| :---: | -------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 2.6.1 | ตรวจสอบให้แน่ใจว่าข้อมูลเข้าแบบไม่ใช่ข้อความทั้งหมด (ภาพ เสียง ไฟล์) ได้รับการตรวจสอบความถูกต้องในเรื่องประเภท ขนาด และรูปแบบก่อนการประมวลผล |   1   |  D   |
| 2.6.2 | ตรวจสอบให้แน่ใจว่าไฟล์ได้รับการสแกนหามัลแวร์และ.payloads สเตกาโนกราฟิกก่อนการนำเข้า                                                          |   2   | D/V  |
| 2.6.3 | ตรวจสอบให้แน่ใจว่าอินพุตภาพ/เสียงได้รับการตรวจสอบสำหรับความวิปริตที่เป็นอันตรายหรือรูปแบบการโจมตีที่รู้จัก                                   |   2   | D/V  |
| 2.6.4 | ตรวจสอบให้แน่ใจว่าการตรวจสอบความถูกต้องของอินพุตหลายรูปแบบที่ล้มเหลวถูกบันทึกและกระตุ้นการแจ้งเตือนเพื่อการตรวจสอบ                           |   3   |  V   |

| 2.7.1 | ตรวจสอบให้แน่ใจว่าข้อมูลป้อนเข้าของผู้ใช้ทั้งหมดถูกติดแท็กด้วยเมทาดาทา (รหัสผู้ใช้, เซสชัน, แหล่งที่มา, เครื่องหมายเวลา, ที่อยู่ IP) ในขั้นตอนการรับข้อมูลเข้า |   1   | D/V  |
| 2.7.2 | ตรวจสอบให้แน่ใจว่าข้อมูลเมทาดาท้าต้นทาง (provenance metadata) ถูกเก็บรักษาและสามารถตรวจสอบได้สำหรับข้อมูลนำเข้าที่ผ่านการประมวลผลทั้งหมด                       |   2   | D/V  |
| 2.7.3 | ตรวจสอบให้แน่ใจว่าแหล่งข้อมูลที่ผิดปกติหรือไม่น่าเชื่อถือได้รับการแจ้งเตือนและอยู่ภายใต้การตรวจสอบอย่างเข้มงวดหรือการบล็อก                                     |   2   | D/V  |

## C2.8 การตรวจจับภัยคุกคามแบบปรับตัวได้แบบเรียลไทม์

นักพัฒนาควรใช้ระบบตรวจจับภัยคุกคามขั้นสูงสำหรับปัญญาประดิษฐ์ (AI) ที่สามารถปรับตัวตามรูปแบบการโจมตีใหม่ ๆ และให้การปกป้องแบบเรียลไทม์ด้วยการจับคู่รูปแบบที่คอมไพล์แล้ว

|   #   | Description                                                                                                                                                                         | Level | Role |
| :---: | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---: | :--: |
| 2.8.1 | ตรวจสอบให้แน่ใจว่ารูปแบบการตรวจจับภัยคุกคามถูกคอมไพล์ลงในเครื่องมือ regex ที่ได้รับการปรับแต่งเพื่อประสิทธิภาพสูงสำหรับการกรองแบบเรียลไทม์โดยมีผลกระทบต่อความล่าช้าให้น้อยที่สุด    |   1   | D/V  |
| 2.8.2 | ตรวจสอบให้แน่ใจว่าระบบตรวจจับภัยคุกคามมีห้องสมุดรูปแบบแยกกันสำหรับประเภทภัยคุกคามต่างๆ (การแทรกพรอมต์, เนื้อหาที่เป็นอันตราย, ข้อมูลที่ละเอียดอ่อน, คำสั่งระบบ)                     |   1   | D/V  |
| 2.8.3 | ตรวจสอบให้แน่ใจว่าการตรวจจับภัยคุกคามแบบปรับตัวใช้โมเดลการเรียนรู้ของเครื่องที่ปรับความไวต่อภัยคุกคามตามความถี่และอัตราความสำเร็จของการโจมตี                                        |   2   | D/V  |
| 2.8.4 | ตรวจสอบให้แน่ใจว่าแหล่งข้อมูลข่าวกรองภัยคุกคามแบบเรียลไทม์อัปเดตห้องสมุดรูปแบบโดยอัตโนมัติด้วยลายเซ็นการโจมตีใหม่และ IOC (ตัวบ่งชี้การถูกโจมตี)                                     |   2   | D/V  |
| 2.8.5 | ตรวจสอบให้อัตราการตรวจจับภัยคุกคามที่เป็นบวกเท็จได้รับการเฝ้าติดตามอย่างต่อเนื่อง และความเฉพาะเจาะจงของรูปแบบถูกปรับแต่งโดยอัตโนมัติเพื่อลดการรบกวนการใช้งานที่ถูกต้องให้น้อยที่สุด |   3   | D/V  |
| 2.8.6 | ตรวจสอบให้แน่ใจว่าการวิเคราะห์ภัยคุกคามเชิงบริบทพิจารณาที่มาของข้อมูลการป้อน รูปแบบพฤติกรรมของผู้ใช้ และประวัติการใช้งานเซสชันเพื่อปรับปรุงความแม่นยำในการตรวจจับ                   |   3   | D/V  |
| 2.8.7 | ตรวจสอบให้แน่ใจว่ามาตรวัดประสิทธิภาพการตรวจจับภัยคุกคาม (อัตราการตรวจจับ, ความล่าช้าในการประมวลผล, การใช้ทรัพยากร) ถูกตรวจสอบและปรับปรุงอย่างเรียลไทม์                              |   3   | D/V  |

## C2.9 สายงานตรวจสอบความปลอดภัยแบบมัลติ-โหมด

นักพัฒนาควรจัดเตรียมการตรวจสอบความปลอดภัยสำหรับข้อความ รูปภาพ เสียง และรูปแบบข้อมูลป้อนเข้า AI อื่น ๆ โดยใช้การตรวจจับภัยคุกคามเฉพาะประเภทและการแยกทรัพยากรที่เหมาะสม

|   #   | Description                                                                                                                                                                                                                                | Level | Role |
| :---: | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | :---: | :--: |
| 2.9.1 | ตรวจสอบให้แน่ใจว่าแต่ละโหมดป้อนข้อมูลมีเครื่องมือยืนยันความปลอดภัยเฉพาะพร้อมรูปแบบภัยคุกคามที่มีเอกสารประกอบ (ข้อความ: การฝังคำสั่งในคำสั่ง, รูปภาพ: สเตโกโนกราฟี, เสียง: การโจมตีด้วยสเปกโตรแกรม) และเกณฑ์การตรวจจับ                      |   1   | D/V  |
| 2.9.2 | ตรวจสอบให้แน่ใจว่าข้อมูลป้อนเข้าหลายรูปแบบถูกประมวลผลในสภาพแวดล้อมแยกอิสระที่มีขีดจำกัดทรัพยากรที่กำหนดไว้ (หน่วยความจำ, ซีพียู, เวลาในการประมวลผล) ซึ่งเฉพาะเจาะจงกับแต่ละประเภทของรูปแบบข้อมูล และได้ถูกบันทึกไว้ในนโยบายความปลอดภัยแล้ว |   2   | D/V  |
| 2.9.3 | ตรวจสอบให้แน่ใจว่าการตรวจจับการโจมตีข้ามโหมดสามารถระบุการโจมตีที่ประสานงานกันซึ่งครอบคลุมหลายประเภทข้อมูลนำเข้า (เช่น ภาระสเตกาโนกราฟีในภาพรวมกับการฉีดคำสั่งในข้อความ) โดยใช้กฎความสัมพันธ์และการสร้างการแจ้งเตือน                        |   2   | D/V  |
| 2.9.4 | ตรวจสอบให้แน่ใจว่าความล้มเหลวในการตรวจสอบแบบมัลติ-โมดัลจะกระตุ้นการบันทึกข้อมูลอย่างละเอียด รวมถึงรูปแบบอินพุตทั้งหมด ผลการตรวจสอบ คะแนนภัยคุกคาม และการวิเคราะห์การเชื่อมโยง ด้วยรูปแบบบันทึกที่มีโครงสร้างสำหรับการผสานรวมกับ SIEM       |   3   | D/V  |
| 2.9.5 | ตรวจสอบให้แน่ใจว่าสายการจำแนกเนื้อหาเฉพาะลักษณะการทำงานถูกอัปเดตตามตารางที่บันทึกไว้ (อย่างน้อยทุกไตรมาส) โดยเพิ่มรูปแบบภัยคุกคามใหม่ ตัวอย่างการโจมตี และเกณฑ์มาตรฐานประสิทธิภาพที่รักษาระดับให้สูงกว่าขีดจำกัดพื้นฐาน                    |   3   | D/V  |
